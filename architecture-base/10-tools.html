Tools Deep Dive

MLOps Tooling — GE, Feast, dbt, BentoML, Seldon

Five tools that complete the production MLOps stack. Great Expectations for data quality, Feast for feature stores, dbt for SQL-based feature engineering, BentoML and Seldon as alternative serving frameworks. Each explained with architecture, code, and when to use it vs alternatives.

Great Expectations — data quality framework
GE
Great Expectations
Define, test, and document data quality expectations as code
Core ideaWrite assertions about your data — "price must be > 0", "null rate must be < 5%", "values must be in this set" — and run them at every ingestion.
Expectation SuiteA collection of expectations for a dataset. Version-controlled JSON file. Run it against any batch of data.
CheckpointA configured run of a Suite against a Data Source. Triggered by Airflow, GitHub Actions, or on any data write.
Data DocsAuto-generated HTML report showing which expectations passed/failed with statistics. Hosted on MinIO S3 or locally.
Data SourceConnects GE to your data — Pandas DataFrame, Parquet on MinIO, SQL table. Abstracts the data access layer.
GE with MinIO — expectation suite code
import great_expectations as gx import pandas as pd, boto3, io # Load data from MinIO s3 = boto3.client("s3", endpoint_url="http://192.168.1.15:30900", aws_access_key_id="minioadmin", aws_secret_access_key="minioadmin") obj = s3.get_object(Bucket="mlops-raw", Key="source=db/date=2025-08-01/train.parquet") df = pd.read_parquet(io.BytesIO(obj["Body"].read())) # Create GE context and validator context = gx.get_context() validator = context.sources.pandas_default.read_dataframe(df) # ── Define expectations ───────────────────────────────── validator.expect_table_row_count_to_be_between(min_value=1000) validator.expect_column_to_exist("MedInc") validator.expect_column_values_to_not_be_null("MedInc", mostly=0.95) validator.expect_column_values_to_be_between( "MedInc", min_value=0, max_value=100) validator.expect_column_values_to_be_between( "HouseAge", min_value=0, max_value=100) validator.expect_column_mean_to_be_between( "MedInc", min_value=2.0, max_value=8.0) # distribution check validator.expect_column_values_to_be_in_set( "OceanProximity", ["NEAR BAY","<1H OCEAN","INLAND","NEAR OCEAN","ISLAND"]) # ── Save suite ────────────────────────────────────────── validator.save_expectation_suite("train_suite") # ── Run checkpoint ────────────────────────────────────── results = validator.validate() if not results["success"]: print("DATA VALIDATION FAILED") print(results.to_json_dict()) exit(1) # halt pipeline — do NOT train on bad data # ── Build and upload Data Docs to MinIO ───────────────── context.build_data_docs() # Serve from: http://192.168.1.15:30900/mlops-artifacts/data_docs/

GE vs Pandera: Pandera is simpler, schema-first, great for quick column-type checks in Python. GE is heavier but has Data Docs, checkpoints, profiling, and a versioned expectation store. Use Pandera for fast in-code validation, GE for a full data quality platform with audit reports.
Feast — feature store
FS
Feast (Feature Store)
Centralised feature management — offline for training, online for serving
The problem it solvesWithout a feature store, every team computes the same features differently. Training pipeline uses one formula for "user average order value in last 7 days". Serving code uses a different formula. Skew guaranteed.
Offline storeHistorical features for training — MinIO/S3 as Parquet files. Supports point-in-time joins so training features are generated as they would have been at the time of each event.
Online storeLow-latency feature serving for real-time inference — Redis, DynamoDB, SQLite. Features are pre-materialized for sub-millisecond lookup at serving time.
Feature viewA definition of features computed from a Data Source. Versioned, registered in Feast's registry. All consumers use the same definition.
EntityThe thing features are computed about — user_id, product_id, house_id. Used as the join key for point-in-time correct feature retrieval.
MaterialisationCopying computed features from the offline store (S3) to the online store (Redis). Run on a schedule to keep online features fresh.
Feast feature definition + retrieval
from feast import (FeatureStore, Entity, FeatureView, Field, FileSource) from feast.types import Float32, Int64 from datetime import timedelta # ── Define data source (MinIO Parquet) ────────────────── house_source = FileSource( path="s3://mlops-processed/features/house_stats.parquet", timestamp_field="event_timestamp", s3_endpoint_override="http://192.168.1.15:30900", ) # ── Define entity (what features are about) ───────────── house = Entity(name="house_id", join_keys=["house_id"]) # ── Define feature view (the computed features) ───────── house_fv = FeatureView( name="house_features", entities=[house], ttl=timedelta(days=7), # online store TTL schema=[ Field(name="med_inc", dtype=Float32), Field(name="house_age", dtype=Float32), Field(name="ave_rooms", dtype=Float32), Field(name="latitude", dtype=Float32), Field(name="longitude", dtype=Float32), ], source=house_source, ) # ── Training: point-in-time correct retrieval ─────────── store = FeatureStore(repo_path="feature_repo/") training_df = store.get_historical_features( entity_df=entity_df, # has house_id + event_timestamp features=[ "house_features:med_inc", "house_features:ave_rooms", ], ).to_df() # ── Serving: online low-latency lookup ────────────────── features = store.get_online_features( features=["house_features:med_inc"], entity_rows=[{"house_id": "h001"}], ).to_dict() # Redis lookup — sub-millisecond
When you need Feast: Multiple models reusing the same features. Multiple teams computing the same feature differently. Low-latency serving requirements needing pre-materialised features in Redis. Not needed for a single-model proof of concept — feature engineering in sklearn Pipeline is enough for that.
dbt — SQL-based feature engineering
dbt
dbt (data build tool)
Transform raw data into features using SQL — with tests, docs, lineage
What it doesRuns SQL transformations in your data warehouse (BigQuery, Redshift, Snowflake, DuckDB). Each transformation is a .sql model file. dbt figures out the dependency order and runs them.
ModelA single SQL SELECT that transforms source data. Saved as a .sql file. dbt wraps it in CREATE TABLE AS SELECT or CTAS. You just write the SELECT.
Ref(){{ ref('other_model') }} — how you reference another model. dbt builds the DAG from these references. Never write raw table names — always ref().
TestsBuilt-in: not_null, unique, accepted_values, relationships. Custom: any SQL assertion. Run after each model build. Equivalent of GE for SQL data.
Lineage graphAuto-generated DAG showing data flow from sources through all transformation models to final tables. Visual documentation.
Incremental modelsOnly process new rows since the last run. Efficient for daily feature computation on large tables.
dbt model — feature computation in SQL
-- models/features/user_order_features.sql -- Computes rolling 7-day order features per user {{ config( materialized = 'incremental', -- only new rows each run unique_key = 'user_id', partition_by = {'field': 'order_date', 'data_type': 'date'} ) }} WITH orders AS ( SELECT * FROM {{ ref('stg_orders') }} -- ref() not raw table {% if is_incremental() %} WHERE order_date > (SELECT MAX(order_date) FROM {{ this }}) {% endif %} ), rolling_features AS ( SELECT user_id, order_date, COUNT(*) OVER( PARTITION BY user_id ORDER BY order_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS orders_7d, AVG(order_value) OVER( PARTITION BY user_id ORDER BY order_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS avg_order_value_7d, DAYS_SINCE(order_date, LAG(order_date) OVER( PARTITION BY user_id ORDER BY order_date)) AS days_since_prev_order FROM orders ) SELECT * FROM rolling_features
# Run dbt dbt run --select user_order_features # one model dbt test --select user_order_features # run tests dbt run --select +user_order_features+ # full upstream DAG dbt docs generate && dbt docs serve # lineage docs
dbt vs Python features: dbt wins for features that are naturally expressed as SQL aggregations over existing warehouse tables — rolling windows, joins, group-bys. Python wins for features requiring ML logic (embeddings, model predictions as features) or when data isn't in a SQL warehouse. Use both: dbt generates the base feature tables, Python pipeline reads them and adds ML-specific transforms.
BentoML — alternative model serving
BML
BentoML
Framework for packaging and serving ML models as containerised services
Core conceptA "Bento" is a packaged unit containing your model + dependencies + serving code. One command builds a Docker image. Deployed anywhere — K8s, AWS Lambda, GCP, local.
RunnerBentoML's abstraction for model inference. Can be local or distributed. Handles batching, concurrency, and GPU sharing automatically.
ServiceA FastAPI-like HTTP service wrapping one or more Runners. Defines endpoints, input/output schemas, middleware.
vs KServeBentoML is more Python-native and easier to start with. KServe is Kubernetes-native with better auto-scaling, canary, and protocol standardisation. BentoML for fast iteration, KServe for production scale.
BentoML service definition
import bentoml import numpy as np from pydantic import BaseModel # ── Save model to BentoML store ───────────────────────── # (after training, instead of / alongside MLflow) bentoml.sklearn.save_model("california_housing", pipe, signatures={"predict": {"batchable": True}}) # ── Define the service (service.py) ───────────────────── runner = bentoml.sklearn.get("california_housing:latest").to_runner() svc = bentoml.Service("housing_service", runners=[runner]) class HouseFeatures(BaseModel): MedInc: float HouseAge: float AveRooms: float AveBedrms: float Population: float AveOccup: float Latitude: float Longitude: float @svc.api(input=bentoml.io.JSON(pydantic_model=HouseFeatures), output=bentoml.io.JSON()) async def predict(house: HouseFeatures) -> dict: arr = np.array([[ house.MedInc, house.HouseAge, house.AveRooms, house.AveBedrms, house.Population, house.AveOccup, house.Latitude, house.Longitude ]]) price = await runner.predict.async_run(arr) return {"price_usd": float(price[0]) * 100_000} # ── Build and serve ───────────────────────────────────── $ bentoml serve service:svc --reload # local dev $ bentoml build # build Bento $ bentoml containerize housing_service:latest # Docker image $ docker run -p 3000:3000 housing_service:latest
Seldon Core — enterprise model serving on K8s
SC
Seldon Core
Production model serving with explainability and outlier detection built in
SeldonDeploymentK8s CRD for deploying models. Similar to KServe InferenceService but with a graph concept — models, routers, combiners, and transformers in one manifest.
Inference graphA pipeline of components: router (A/B test) → model → outlier detector → explainer → logger. All defined in one YAML, all running in the same pod group.
ExplainerBuilt-in SHAP/LIME explainability endpoint. GET /v1/models/x:explain returns per-feature attribution. KServe requires separate setup. Seldon has it by default.
Alibi DetectSeldon's companion library for outlier detection, concept drift detection, and adversarial example detection. Runs inline in the inference graph.
vs KServeSeldon has richer inference graphs and built-in explainability. KServe has better auto-scaling and is the CNCF standard. Both run on K8s. Seldon is favoured in regulated industries (finance, healthcare).

Tool decision matrix — which to use when
Data validationQuick schema checks in Python → Pandera. Full audit reports + checkpoint history → Great Expectations. SQL table tests → dbt tests.
Feature engineeringPython ML features (scaling, OHE, embeddings) → sklearn Pipeline. SQL aggregations on warehouse data → dbt. Reuse across many models + teams → Feast.
Experiment trackingSimple, self-hosted → MLflow. Managed, collaboration, reports → Weights & Biases. In Kubeflow ecosystem → KFP + MLMD.
OrchestrationComplex ETL + ML + cross-system → Airflow. Pure ML, K8s-native → Kubeflow Pipelines. Simple code pipelines → GitHub Actions.
Model servingK8s, production, auto-scale, canary → KServe. Fast dev, any platform → BentoML. Explainability + inference graph → Seldon Core. Simple FastAPI → FastAPI + uvicorn.
Drift monitoringOpen-source, self-hosted → Evidently. Streaming, real-time → Alibi Detect (Seldon). Enterprise managed → Arize AI, Fiddler.
Feature storeOpen-source → Feast. Managed → Tecton, Hopsworks. In-house → Redis + DVC offline.
Home lab (your stack)DVC + MLflow + MinIO + sklearn Pipeline + KServe + Evidently + GitHub Actions. This covers 80% of what Flipkart/Swiggy interviewers want to hear. Airflow and KFP are bonus points.
