Apache Airflow — DAGs, Operators, MLOps Pipelines

Airflow is the scheduler and orchestrator that glues every MLOps stage together. It replaces cron jobs with a visual, retryable, dependency-aware pipeline. This page covers DAG anatomy, every operator type, and three production MLOps DAG patterns — ingestion, training, and retraining.

What Airflow is and is not
What Airflow IS
A workflow scheduler. It defines when tasks run, in what order, handles retries on failure, and gives you a visual UI to monitor every run. It is the conductor of your ML orchestra.
What Airflow is NOT
Not a compute engine. Not a data streaming platform. Not a model training framework. Airflow just calls your Python, Spark, SQL, or Kubernetes jobs. The actual work happens elsewhere.
Why it beats cron
Cron has no retries, no dependency management, no UI, no alerting. Airflow gives all of these. If ingestion fails, training never starts. If training fails, deployment never happens. Cron can't do this.
SRE analogy: Airflow is like a Runbook that executes itself. Each step has preconditions, timeouts, retries, and escalation paths — exactly what you'd want in a production incident runbook, but automated.
Core concepts
DAGDirected Acyclic Graph. The definition of your workflow — which tasks exist and in what order they must run. A Python file in the dags/ folder. Acyclic = no loops, tasks don't circle back.
TaskA single unit of work inside a DAG. Implemented by an Operator. Each task has its own state: queued, running, success, failed, skipped, upstream_failed.
OperatorThe class that defines what a task does. PythonOperator runs Python. BashOperator runs shell. KubernetesPodOperator runs a K8s pod. You pick the right operator for each task.
DAG RunOne execution of the entire DAG for a specific logical_date. Airflow can backfill (run DAG for past dates you missed) — powerful for re-processing historical data.
logical_dateThe date the DAG run is meant to process, not when it actually ran. A DAG scheduled daily at midnight processes data for yesterday — logical_date is yesterday.
XComCross-communication. Tasks push small values (e.g. a DVC hash, a model run_id) and downstream tasks pull them. Not for passing large data — use MinIO for that.
ConnectionStored credentials in Airflow's encrypted database. S3 connection (MinIO), Postgres connection (MLflow DB), HTTP connection (MLflow API). Never hardcode secrets.
VariableKey-value config stored in Airflow. E.g. MLFLOW_URI, MINIO_BUCKET. Change config without editing DAG code.
SchedulerThe Airflow component that reads DAG files, triggers runs at schedule time, and assigns tasks to workers.
ExecutorHow Airflow runs tasks. LocalExecutor: same machine. CeleryExecutor: distributed workers. KubernetesExecutor: each task runs as a K8s pod — ideal for MLOps.
SensorA special operator that waits for an external condition — file arrives in MinIO, HTTP endpoint returns 200, another DAG completes. Blocks until condition is met then proceeds.
SLAService Level Agreement per task. If a task takes longer than the defined SLA, Airflow sends a miss alert — useful for catching slow ETL or training jobs.

Operator types — when to use which
PythonOperatorRuns any Python callable. Most flexible. Good for lightweight tasks — validation, API calls, MLflow promotion. NOT for heavy compute — blocks a worker thread.
BashOperatorRuns shell commands. dvc pull, dvc push, git commands. Simple, no Python wrapper needed.
KubernetesPodOperatorBest for MLOps. Each task runs in its own K3s pod with custom image, CPU/memory limits, environment variables. Heavy training, feature engineering, batch inference. Clean isolation.
S3KeySensor / MinIOSensorWaits until a specific key exists in MinIO. Use to block training until upstream ETL has written its output. Pokes on an interval and releases the worker slot between checks (reschedule mode).
HttpSensorWaits until an HTTP endpoint returns a success code. Used to check if MLflow, KServe, or an external API is healthy before proceeding.
ExternalTaskSensorWaits for a task or entire DAG in a different DAG to succeed. Use when the training DAG should wait for the ingestion DAG to complete first.
SparkSubmitOperatorSubmits a Spark job. For large-scale feature engineering that can't run in a single Python process.
TriggerDagRunOperatorTriggers another DAG programmatically from within a DAG. Use to chain ingestion → training → deployment DAGs without building one monolithic DAG.
DockerOperatorRuns a Docker container. Alternative to KubernetesPodOperator for non-K8s environments.
EmailOperatorSends an email. Often used in on_failure_callback for alerting instead of as a standalone task.
Task dependency patterns
Pattern 1 — Linear chain (most common for training)
# A → B → C → D ingest >> validate >> featurize >> train >> promote # If validate fails, featurize never starts. # If train fails, promote never runs. # Retries happen at the failed task only.
Pattern 2 — Fan-out / Fan-in (parallel feature groups)
# Ingest splits into 3 parallel feature jobs, then merge ingest >> [feat_numeric, feat_categorical, feat_temporal] [feat_numeric, feat_categorical, feat_temporal] >> merge_features >> train # All 3 feature tasks run in parallel. # Merge only starts when ALL 3 succeed. # Saves wall-clock time for independent workloads.
Pattern 3 — Conditional branching (BranchPythonOperator)
from airflow.operators.python import BranchPythonOperator def check_drift(**context): psi = get_latest_psi_score() return "trigger_retrain" if psi > 0.2 else "skip_retrain" branch = BranchPythonOperator( task_id="check_drift", python_callable=check_drift, ) # Only one branch executes based on PSI score branch >> [trigger_retrain, skip_retrain]
Pattern 4 — Passing data between tasks with XCom
def run_training(**context): with mlflow.start_run() as run: # ... training logic ... # Push run_id to XCom so promote task can use it context["ti"].xcom_push(key="mlflow_run_id", value=run.info.run_id) def promote_model(**context): run_id = context["ti"].xcom_pull( task_ids="train_model", key="mlflow_run_id") # Promote the specific run_id client.transition_model_version_stage( name="california-housing", version=run_id, stage="Production")

Airflow vs alternatives — when to use which
Apache AirflowBest for complex ETL with many dependencies. Large community, rich operator library (S3, GCS, Spark, Kubernetes, dbt). Steep setup. Used at Flipkart, Walmart, Swiggy scale. Your interview target companies run Airflow.
PrefectModern Python-native alternative. Dynamic workflows (loops, conditionals in plain Python). Easier to unit-test than Airflow. Prefect Cloud for observability. Growing adoption in mid-market.
DagsterAsset-centric orchestration. You define data assets, not tasks. Dagster figures out the order. Excellent observability and lineage built-in. Gaining traction at data-mature startups.
GitHub ActionsFine for code-triggered ML pipelines (git push → train → deploy). Not designed for data-triggered scheduling or complex dependency graphs. Complements Airflow, doesn't replace it.
Kubeflow PipelinesML-specific. Containers as steps. Native K8s, built for ML workflows. Better for Kubeflow ecosystem (Katib, KServe). Airflow is more general. See next page for full comparison.

Common Airflow failures and fixes
DAG import errorSyntax or import error in a DAG file — Airflow skips the whole file silently. Always check Admin → DAG Import Errors in the UI. Fix the Python error first.
Zombie tasksA task appears running but the worker died. Airflow detects after zombie_task_threshold (5 min default). Click "Clear" to re-queue the task.
max_active_runs overflowMultiple DAG runs piling up — the new run doesn't wait for the previous to finish. Set max_active_runs=1 to serialise runs.
XCom size limitXCom stores in the Airflow DB — max ~48KB. Never push a DataFrame or model through XCom. Push a MinIO key or MLflow run_id instead.
Sensor timeoutData never arrived in MinIO → sensor waits until timeout then fails the DAG. Add alerting on sensor timeout. Investigate upstream ETL.
Catchup disasterSetting catchup=True with a daily DAG and a start_date 1 year ago = 365 simultaneous runs triggered. Always set catchup=False unless explicitly backfilling.
