Kubeflow Pipelines — KFP, Katib, Serving

Kubeflow is a complete ML platform built on top of Kubernetes. Kubeflow Pipelines (KFP) orchestrates ML workflows as containerised components. Katib does automated hyperparameter tuning. KServe handles model serving. This page covers every component with full code examples and a direct comparison with Airflow.

Kubeflow — the complete ML platform
Kubeflow Pipelines (KFP)
ML workflow orchestration. Each step is a Docker container. Steps are chained into a pipeline YAML. Runs on K8s natively. The Argo Workflows engine executes the DAG underneath.
Katib
Automated hyperparameter tuning and NAS. Defines a search space (learning_rate: 0.01–0.1, n_estimators: 50–500), runs parallel training jobs, picks the best. Bayesian, grid, random, evolutionary strategies.
KFP + KServe
Kubeflow's serving layer is KServe — the same KServe you already run. Kubeflow adds UI management of InferenceServices on top. Canary rollout, traffic management, model monitoring all in one platform.
Key insight: Kubeflow is ML-first. Airflow is general-purpose data engineering that includes ML. If your entire stack is ML, Kubeflow gives a tighter integration. If you have mixed ETL + ML + analytics, Airflow wins on breadth.
KFP core concepts
ComponentThe atomic building block. A Python function or a containerised script that does one thing (featurize, train, evaluate). Described by its inputs, outputs, and the Docker image it runs in.
PipelineA graph of Components with dependencies. Defined in Python using the @pipeline decorator. Compiled to a pipeline YAML that KFP executes on K8s.
ExperimentA grouping of pipeline runs. Analogous to an MLflow experiment. Allows comparing different runs of the same pipeline.
RunOne execution of a pipeline for a specific set of inputs. Each run creates Argo Workflow pods on K8s. Logs and artifacts stored in MinIO.
ArtifactData passed between components via typed, tracked paths. Dataset, Model, Metrics, HTML artifacts — all versioned by KFP's ML Metadata (MLMD) store.
MLMDML Metadata store. KFP's built-in lineage tracking. Records which artifact was produced by which component execution. Stored in MySQL on K8s.
Recurring RunA pipeline scheduled on a cron expression. Alternative to Airflow scheduling for ML pipelines that don't need complex cross-system dependencies.
Argo WorkflowsThe underlying K8s workflow engine KFP is built on. Each KFP step becomes an Argo Workflow step — a K8s pod. You can inspect raw Argo Workflows for debugging.
Defining KFP components — two ways
Method 1 — @component decorator (lightweight Python)
from kfp import dsl from kfp.dsl import Input, Output, Dataset, Model, Metrics @dsl.component( base_image="python:3.11-slim", packages_to_install=["pandas", "scikit-learn", "mlflow"] ) def train_model( train_data: Input[Dataset], # typed input — KFP tracks lineage val_data: Input[Dataset], model_out: Output[Model], # typed output artifact metrics: Output[Metrics], # shown in KFP UI n_estimators: int = 100, mlflow_uri: str = "http://192.168.1.15:30500", ): import pandas as pd, mlflow, pickle from sklearn.ensemble import RandomForestRegressor from sklearn.metrics import mean_squared_error X_train = pd.read_parquet(train_data.path) X_val = pd.read_parquet(val_data.path) y_train = X_train.pop("target") y_val = X_val.pop("target") mlflow.set_tracking_uri(mlflow_uri) with mlflow.start_run(): rf = RandomForestRegressor(n_estimators=n_estimators) rf.fit(X_train, y_train) rmse = mean_squared_error(y_val, rf.predict(X_val)) ** 0.5 mlflow.log_metric("val_rmse", rmse) # Save model to KFP output path with open(model_out.path, "wb") as f: pickle.dump(rf, f) # Log metrics to KFP UI metrics.log_metric("val_rmse", rmse)
Method 2 — ContainerSpec (full Docker image, any language)
from kfp import dsl # Use a pre-built image — any language, any framework @dsl.container_component def run_spark_featurization(date: str) -> dsl.ContainerSpec: return dsl.ContainerSpec( image="ghcr.io/koyilada-rishi/spark-feature:latest", command=["spark-submit"], args=[ "--master", "k8s://https://kubernetes:443", "src/featurize.py", "--date", date, ], ) # Useful when: # - You need Spark/Java/Go/R — not just Python # - Heavy dependencies that bloat the decorator image # - Reusing an existing Docker image
Full KFP pipeline — data to deployed model
pipeline.py — complete end-to-end KFP pipeline with conditions
from kfp import dsl, compiler from kfp.dsl import Input, Output, Dataset, Model, Metrics @dsl.pipeline( name="mlops-california-housing", description="End-to-end training and deployment for housing price model", ) def mlops_pipeline( date: str = "2025-08-01", n_estimators: int = 100, rmse_threshold: float = 0.50, ): # Step 1: Validate raw data in MinIO validate_task = validate_data(date=date) # Step 2: Featurize — depends on validate succeeding featurize_task = featurize( date=date, ).after(validate_task) # Step 3: Train — takes featurize output as input train_task = train_model( train_data=featurize_task.outputs["train_data"], val_data=featurize_task.outputs["val_data"], n_estimators=n_estimators, ) # Step 4: Evaluate — conditional promotion eval_task = evaluate_model( model=train_task.outputs["model_out"], test_data=featurize_task.outputs["test_data"], ) # Step 5: Promote only if RMSE below threshold with dsl.If(eval_task.outputs["rmse"] < rmse_threshold): promote_task = promote_model( model=train_task.outputs["model_out"], model_name="california-housing", ) # Resource customisation per step train_task.set_memory_request("4G") train_task.set_cpu_request("2") train_task.set_gpu_limit("1") # optional: GPU training # Compile to YAML compiler.Compiler().compile( pipeline_func=mlops_pipeline, package_path="pipeline.yaml" ) # Submit run via Python client import kfp client = kfp.Client(host="http://192.168.1.15:8080") client.create_run_from_pipeline_package( pipeline_file="pipeline.yaml", arguments={"date": "2025-08-01", "n_estimators": 200}, run_name="housing-run-20250801", experiment_name="california-housing", )
Katib — automated hyperparameter tuning
What Katib does
Katib runs dozens of training jobs in parallel, each with different hyperparameters from your search space. It learns from each run (Bayesian optimisation) to suggest better parameters next. Similar to Optuna but native K8s — each trial is a full K8s pod with GPU support.
Katib vs Optuna: Optuna runs in a single process on your training machine. Katib runs each trial as a separate K8s pod — massively parallel. Use Katib when you have a K8s cluster and GPU budget. Use Optuna when you want simplicity.
Katib Experiment YAML
apiVersion: kubeflow.org/v1beta1 kind: Experiment metadata: name: housing-hparam-search namespace: kubeflow spec: objective: type: minimize goal: 0.40 objectiveMetricName: val_rmse algorithm: algorithmName: bayesianoptimization # or random, grid, cmaes parallelTrialCount: 4 # 4 pods in parallel maxTrialCount: 50 maxFailedTrialCount: 5 parameters: - name: n_estimators parameterType: int feasibleSpace: {min: "50", max: "500"} - name: max_depth parameterType: int feasibleSpace: {min: "3", max: "20"} - name: learning_rate parameterType: double feasibleSpace: {min: "0.01", max: "0.3"} trialTemplate: trialSpec: apiVersion: batch/v1 kind: Job spec: template: spec: containers: - name: train image: ghcr.io/koyilada-rishi/mlops-train:latest command: - python - src/train.py - --n_estimators=${trialParameters.n_estimators} - --max_depth=${trialParameters.max_depth}
Airflow vs Kubeflow Pipelines — full comparison
Primary use caseAirflow: General data engineering + ML. Complex multi-system workflows (DB → Spark → S3 → ML → Slack).
KFP: Pure ML pipelines. Components are always containers. Tightly coupled to K8s.
Step definitionAirflow: Python function or any Operator (Python, Bash, Spark, K8s, HTTP, email...).
KFP: Always a Docker container. @component or ContainerSpec. No non-container steps.
Data passingAirflow: XCom for small values. MinIO paths for large data. No automatic lineage.
KFP: Typed artifacts (Dataset, Model, Metrics) with automatic MLMD lineage tracking.
Lineage trackingAirflow: Manual (log to MLflow yourself).
KFP: Automatic via MLMD store — every artifact linked to every run that produced or consumed it.
Hyperparameter tuningAirflow: Integrate Optuna or Hyperopt yourself within a task.
KFP: Katib built-in — native parallel K8s tuning jobs.
UI qualityAirflow: Strong — DAG graph, Gantt chart, task logs, run history.
KFP: Strong for ML — pipeline graphs, artifact visualisation, Jupyter notebooks inline.
SchedulingAirflow: Excellent — cron, external triggers, sensors, complex schedule_interval.
KFP: Basic recurring runs. Airflow often manages KFP trigger timing from outside.
Setup complexityAirflow: Moderate — Helm chart, Postgres, S3 for logs, Redis for Celery.
KFP: High — Argo Workflows, MLMD, MySQL, MinIO, Katib all on K8s.
Who uses whatAirflow: Flipkart, Walmart, Swiggy for data pipelines.
KFP: Google AI Platform, ML teams wanting an end-to-end ML-native platform.
Best combined useAirflow schedules KFP runs. Airflow handles ingestion (ETL, sensors, alerts) and then calls KFP when data is ready for ML processing. Each tool does what it's best at.
KFP on K3s — setup
# Install KFP standalone (lighter than full Kubeflow) kubectl apply -k "github.com/kubeflow/pipelines/manifests/kustomize/cluster-scoped-resources?ref=2.2.0" kubectl apply -k "github.com/kubeflow/pipelines/manifests/kustomize/env/platform-agnostic-pns?ref=2.2.0" # Port-forward KFP UI (or use NodePort) kubectl port-forward svc/ml-pipeline-ui 8080:80 -n kubeflow # Access at http://192.168.1.15:8080 # Submit a pipeline run via CLI pip install kfp kfp run create --pipeline-name "housing" --run-name "run-001" \ --pipeline-package-path pipeline.yaml
Note: Full Kubeflow needs 16GB+ RAM. KFP standalone needs ~6GB. On your Proxmox home lab, allocate at least 8GB to the K3s VM running KFP. MLflow + MinIO + KFP + Prometheus together need ~10GB total.
