Feature engineering in MLOps

From raw data to model-ready numbers — transformations, encoding, selection, pipeline, and production integration.

Feature engineering — turning raw data into model language

A model cannot learn from raw data directly. It learns from numbers arranged so patterns are visible. Feature engineering is the translation layer between real-world data and numbers the model can reason over.

Why it matters more than the algorithm
A good random forest on great features beats a neural network on raw data almost every time. The algorithm learns patterns that exist in the feature space. If the feature space is bad, no algorithm can save it.
What goes wrong without it
Model sees "Mumbai" as a string — can't process it. Sees null — crashes or silently zeros. Sees 2019-08-14 as a string — misses seasonality, recency, day-of-week. The model is blind to structure that's actually there.
The 80/20 rule of ML
Feature engineering accounts for roughly 80% of performance gain. Switching RandomForest to XGBoost might get you 2%. Adding the right temporal feature might get you 15%.
Types of features you will create
Numeric transformsScale, normalise, log, bin, polynomial
Categorical encodingOHE, ordinal, target, embedding
Temporal featuresDay, hour, lag, rolling average
Text / NLPTF-IDF, embeddings, sentiment score
Interaction featuresFeature A × Feature B, ratios
Domain featuresBusiness logic baked into a number
The contract a model expects
Every model expects a fixed-width numeric matrix — the same number of columns, in the same order, with no nulls, every time it runs. Feature engineering enforces this contract. It is the interface between raw reality and model assumptions.
In sklearn a Pipeline enforces this contract at both training and serving time. One object, one fit, one predict. Same transforms guaranteed.

Numeric transformations

Raw numbers are rarely in the right shape for a model. Scales differ wildly, distributions are skewed, outliers dominate. Click a transform to see the before/after and when to use it.

Standardisation (z-score)
Mean=0, Std=1. For scale-sensitive algorithms: linear regression, SVM, KNN, PCA.
Min-max normalisation
Squeezes to [0,1]. Good for neural networks and when bounds are known.
Log transform (log1p)
Compresses right-skewed distributions. Classic for price, income, counts.
Binning / discretisation
Age 34 → "25-35" bucket. Removes outlier sensitivity for tree models.
Clipping outliers
Cap at p1/p99. Prevents one bad row from dominating the scaler.
Polynomial features
A², A×B. Captures non-linear interactions for linear models.
Imputation — handling nulls (must happen BEFORE scaling)
from sklearn.impute import SimpleImputer # Strategy choice: # median → robust to outliers — use for skewed numeric cols # mean → only for normally distributed cols # most_frequent → for categoricals # constant → when "missing" is itself a signal imp = SimpleImputer(strategy="median") imp.fit(X_train) # learns median from training data ONLY X_train = imp.transform(X_train) X_val = imp.transform(X_val) # uses training median — not val median # Advanced: binary "was_null" indicator — the absence is itself a signal from sklearn.impute import MissingIndicator indicator = MissingIndicator() X_missing_flags = indicator.fit_transform(X_train_raw) # now model knows WHICH rows had nulls

Categorical encoding

Models need numbers. Categories — city names, product types, user segments — must be converted. Wrong encoding = lost signal or exploded dimensionality.

One-hot encoding (OHE) — low cardinality (<20 values)
Creates one binary column per category. "city" with 3 values → 3 columns.
Before
city Mumbai Delhi Mumbai
→
After OHE
Mumbai Delhi 1 0 0 1 1 0
1000 cities → 1000 columns → sparse matrix, slow training, overfitting.
from sklearn.preprocessing import OneHotEncoder ohe = OneHotEncoder( handle_unknown="ignore", # unseen at serve → all zeros sparse_output=False ) ohe.fit(X_train[["city"]])
Target encoding — high cardinality (20–1000 values)
Replace category with the mean target value for that category. Compact, powerful.
Before
city Mumbai Delhi Pune
→
After target enc
city_enc 4.21 3.87 2.94
Must compute target mean on training fold ONLY — never on full dataset. Otherwise it leaks the target into features (data leakage).
from category_encoders import TargetEncoder te = TargetEncoder(smoothing=10) # smoothing = prevent overfitting on rare cats te.fit(X_train["city"], y_train)
Ordinal encoding — ordered categories
When order matters: Low=1, Medium=2, High=3. Tree models handle this well. Do NOT use for unordered categories — it implies a false ordering.
from sklearn.preprocessing import OrdinalEncoder oe = OrdinalEncoder(categories=[[ "low", "medium", "high" # explicit order ]]) oe.fit(X_train[["risk_level"]])
Embeddings — very high cardinality (1000+ values)
User ID, product ID, zip code. Train a dense vector. Captures similarity: nearby embeddings = similar behaviour.
import torch.nn as nn embed = nn.Embedding( num_embeddings=n_cities, # vocab size embedding_dim=16 # dense vector dimension ) # city_id=42 → 16-dim float vector # Similar cities cluster in embedding space
Decision guide: <20 unique values → OHE. 20–1000 → target encoding. 1000+ → embeddings. Has natural order → ordinal. For tree models: ordinal or target encoding usually beats OHE.

Temporal features — extracting signal from timestamps

A raw timestamp "2025-08-14 14:32:00" is useless to a model. But the hour, day-of-week, days-since-last-purchase, and 7-day rolling average are extremely powerful. Time is rich with patterns — seasonality, recency, periodicity.

Point-in-time features — extracted from the timestamp
import pandas as pd, numpy as np df["ts"] = pd.to_datetime(df["ts"]) # Calendar features df["hour"] = df["ts"].dt.hour df["day_of_week"] = df["ts"].dt.dayofweek # 0=Mon df["month"] = df["ts"].dt.month df["is_weekend"] = df["ts"].dt.dayofweek >= 5 df["quarter"] = df["ts"].dt.quarter # Cyclical encoding — avoids the 23→0 hour jump df["hour_sin"] = np.sin(2 * np.pi * df["hour"] / 24) df["hour_cos"] = np.cos(2 * np.pi * df["hour"] / 24) # Now hour 23 and hour 0 are close in feature space
Recency and lag features
# Recency: how long since the last event? df = df.sort_values("ts") df["days_since_last_purchase"] = ( df["ts"] - df.groupby("user_id")["ts"].shift(1) ).dt.days # Lag features: value N periods ago df["sales_lag_1d"] = df["sales"].shift(1) df["sales_lag_7d"] = df["sales"].shift(7) # Rolling statistics: trend over a window # IMPORTANT: shift(1) before rolling to exclude current row df["rolling_7d_mean"] = ( df["sales"].shift(1).rolling(window=7).mean() ) df["rolling_7d_std"] = ( df["sales"].shift(1).rolling(window=7).std() )
Critical rule — event_time not ingest_time
Always compute temporal features from when the event happened, not when it was ingested. A sale at 11pm ingested at 2am must use 11pm for day_of_week. Ingest time causes systematic error.
Time-series split — never shuffle
Train must be on past, validation on future. Never shuffle rows across time. Otherwise lag features leak future values into training — you're training on "data from tomorrow."
Cyclical encoding — why it matters
Hour 23 and hour 0 are 1 hour apart in reality but 23 apart as raw numbers. Sin/cos encoding wraps the circle so 23 and 0 are close in feature space. Same for month, day-of-week.
Biggest temporal mistake: rolling average that includes the current row. The value you're predicting is inside the window. Always .shift(1) before .rolling() to exclude the current row.

Text and embedding features

Text must become a fixed-length numeric vector before entering a model. The method you choose determines how much semantic meaning is preserved.

TF-IDF — bag of words with importance weighting
TF = how often a word appears in this document. IDF = penalises words that appear everywhere (like "the"). Sparse, fast, interpretable, no GPU needed.
from sklearn.feature_extraction.text import TfidfVectorizer tfidf = TfidfVectorizer( max_features=5000, # keep top 5000 words ngram_range=(1,2), # unigrams + bigrams min_df=5, # ignore words in fewer than 5 docs sublinear_tf=True # log(tf) — dampens high frequency ) tfidf.fit(X_train["description"]) X_tfidf = tfidf.transform(X_train["description"]) # sparse matrix: shape (n_rows, 5000)
Sentence embeddings — semantic meaning preserved
Pre-trained model converts text to a dense 384/768-dim vector. "Good product" and "Excellent item" end up close in vector space. Far better semantic understanding.
from sentence_transformers import SentenceTransformer model = SentenceTransformer("all-MiniLM-L6-v2") embeddings = model.encode( df["review_text"].tolist(), batch_size=64 ) # shape: (n_rows, 384) # "Great product" ≈ "Excellent item" in this space # Or locally with Ollama — no API cost import ollama resp = ollama.embed(model="nomic-embed-text", input="Great product") vector = resp["embeddings"][0] # 768-dim
Derived text features (free, fast)
df["text_len"] = df["text"].str.len() df["word_count"] = df["text"].str.split().str.len() df["has_question"] = df["text"].str.contains("?") df["excl_count"] = df["text"].str.count("!") df["caps_ratio"] = ( df["text"].str.count(r"[A-Z]") / df["text"].str.len() )

TF-IDFClassification, tabular ML, interpretability needed
EmbeddingsSemantic similarity, search, clustering, LLM input
DerivedAlways — they're free and often surprisingly predictive

Feature selection — less is often more

More features is not better. Irrelevant features add noise, slow training, cause overfitting, and make models harder to debug. Feature selection finds which features actually contribute signal.

Filter methods — fast, model-agnostic
Score each feature independently with a statistical test. No model training needed. Fast enough to run on every dataset version.
from sklearn.feature_selection import ( SelectKBest, f_classif, mutual_info_classif) # F-test: linear correlation with target sel = SelectKBest(f_classif, k=20) sel.fit(X_train, y_train) # Mutual information: catches non-linear deps import pandas as pd mi = mutual_info_classif(X_train, y_train) mi_scores = pd.Series(mi, index=feature_names) top20 = mi_scores.nlargest(20).index.tolist()
Tree-based feature importance — free from the model
import pandas as pd from sklearn.ensemble import RandomForestClassifier rf = RandomForestClassifier(n_estimators=100) rf.fit(X_train, y_train) importances = pd.Series( rf.feature_importances_, index=feature_names ).sort_values(ascending=False) # Drop features below a threshold keep = importances[importances > 0.01].index X_train_sel = X_train[keep] # Log to MLflow for audit mlflow.log_dict( importances.to_dict(), "feature_importances.json")
SHAP — best for understanding AND selecting
import shap, pandas as pd explainer = shap.TreeExplainer(model) shap_values = explainer.shap_values(X_val) mean_shap = pd.Series( abs(shap_values).mean(axis=0), index=feature_names ).sort_values(ascending=False) keep = mean_shap[mean_shap > 0.001].index # SHAP also tells you the DIRECTION of impact # High MedInc → pushes price up (positive SHAP)
What to drop and what to keep
Drop Zero-variance features (same value in every row)
Drop Features with >90% nulls
Drop Duplicate columns (correlation = 1.0)
Drop Raw IDs, raw timestamps

Keep Low-importance features with zero serving cost
Keep Domain features even if weak — interpretability value
Never select features on the full dataset before splitting. Selection itself can leak information. Fit your selector on training data only, then apply to val and test.

Putting it all together — the sklearn Pipeline

Every transform must run identically at training and serving time. The sklearn Pipeline + ColumnTransformer guarantees this. One object. One fit. One predict. Forever.

Complete production pipeline — numeric + categorical + model
from sklearn.pipeline import Pipeline from sklearn.compose import ColumnTransformer from sklearn.impute import SimpleImputer from sklearn.preprocessing import StandardScaler, OneHotEncoder from sklearn.ensemble import RandomForestRegressor import mlflow num_features = ["MedInc", "HouseAge", "AveRooms", "AveOccup"] cat_features = ["OceanProximity"] num_pipe = Pipeline([ ("impute", SimpleImputer(strategy="median")), ("scale", StandardScaler()), ]) cat_pipe = Pipeline([ ("impute", SimpleImputer(strategy="most_frequent")), ("encode", OneHotEncoder(handle_unknown="ignore")), ]) preprocessor = ColumnTransformer([ ("num", num_pipe, num_features), ("cat", cat_pipe, cat_features), ]) full_pipe = Pipeline([ ("preprocess", preprocessor), ("model", RandomForestRegressor(n_estimators=100, random_state=42)), ]) full_pipe.fit(X_train, y_train) # fit on training data ONLY val_preds = full_pipe.predict(X_val) # same transforms, guaranteed test_preds = full_pipe.predict(X_test) with mlflow.start_run(): mlflow.sklearn.log_model(full_pipe, "model") # Frozen inside: imputer.statistics_, scaler.mean_, ohe.categories_ # Serving: full_pipe.predict(X_new) — no re-fitting. Ever.
What gets frozen inside the .pkl
SimpleImputer.statistics_ — median per col from X_train

StandardScaler.mean_ and scale_ — from imputed X_train

OneHotEncoder.categories_ — all cats seen in training

RandomForest.estimators_ — all 100 trained trees

All live inside the single .pkl. Load it at serving, call .predict(). Nothing recomputed.
Custom transforms in the pipeline
from sklearn.base import BaseEstimator, TransformerMixin import numpy as np class CyclicalEncoder(BaseEstimator, TransformerMixin): def __init__(self, period): self.period = period def fit(self, X, y=None): return self # stateless — nothing to learn def transform(self, X): sin = np.sin(2 * np.pi * X / self.period) cos = np.cos(2 * np.pi * X / self.period) return np.hstack([sin, cos]) hour_pipe = Pipeline([ ("cyclical", CyclicalEncoder(period=24)) ]) # Plug into ColumnTransformer like any other sub-pipeline

Feature engineering in the MLOps pipeline

Feature engineering has a precise position: after raw validation, before training. It must produce DVC-tracked, reproducible artifacts that flow into the model registry.

Feature engineering as a DVC stage
# dvc.yaml stages: featurize: cmd: python src/features.py deps: - data/raw/train.parquet # input - src/features.py # code change → re-run params: - params.yaml: - feature.max_na_pct - feature.scale_method outs: - data/processed/features.parquet # dvc repro only re-runs if train.parquet # OR features.py changed. Otherwise uses cache.
Feature drift monitoring in production
After deployment, Evidently monitors whether feature distributions at serving time match what the model was trained on.
# What Evidently checks for features: # PSI per feature — distribution shift # Null rate change — imputation now fills more # New unseen categories — OHE → all zeros silently # Out-of-range numerics — clipping fires silently # # All drift alerts → Prometheus → Grafana # → retrain trigger via GitHub Actions
