MLOps Glossary — 100+ Terms

Complete terminology for every concept in the MLOps pipeline. Search or filter by category. Study this before interviews — knowing the right vocabulary shows depth of understanding.

Data & Storage
MinIOData
Open-source, S3-compatible object storage. Acts as the mandatory first landing zone for all raw data before any ML processing. Your home lab runs it at 192.168.1.15:30900.
DVCData
Data Version Control. Tracks dataset versions via MD5 hash pointers stored in Git. Actual file bytes live in MinIO. dvc pull downloads the exact version, dvc push uploads it.
ETLData
Extract, Transform, Load. The process of pulling data from a source (Extract), converting it to a suitable format (Transform), and writing it to storage (Load). DB → parquet → MinIO.
Data lakeData
A storage repository holding vast amounts of raw data in its native format until needed. MinIO's mlops-raw/ bucket acts as the data lake in your architecture.
ParquetData
Columnar storage format for big data. Highly compressed, supports predicate pushdown (read only the columns you need). The standard format for ML datasets stored in MinIO.
KafkaData
Distributed event streaming platform. Produces and consumes high-throughput event data. In MLOps: streaming source that gets micro-batched and written to MinIO with event-time partitioning.
Data lineageData
The ability to trace any model or prediction back to the exact dataset, code version, and configuration that produced it. Achieved via MLflow run_id + DVC hash + git SHA.
Schema registryData
Centralised store of data schemas. Detects when a source system changes its column structure. Alerts pipeline before bad data corrupts feature engineering.
Feature storeData
Centralised system managing ML features. Offline store (MinIO/Parquet) for training data. Online store (Redis) for low-latency serving. Feast and Hopsworks are popular implementations.
Idempotent writeData
A write operation that produces the same result when applied multiple times. Critical for ETL jobs — if the job re-runs due to failure, it should not duplicate data in MinIO.
Event time vs ingest timeData
Event time = when the event happened (a sale at 11pm). Ingest time = when it arrived in your system (2am the next day). All temporal features must use event time for correct day_of_week, hour etc.
Data versioningData
Tracking which exact version of a dataset was used for each model training run. DVC provides this via .dvc pointer files committed to Git alongside the code.
Feature Engineering
StandardisationModel
Transform a numeric feature to have mean=0 and std=1. (x - mean) / std. Required for algorithms sensitive to scale: SVM, KNN, linear regression, PCA, neural networks.
Min-max normalisationModel
Scale features to a fixed range [0,1]. (x - min) / (max - min). Preserves zero. Sensitive to outliers — one extreme value compresses all others. Good for neural networks and images.
One-hot encoding (OHE)Model
Convert a categorical feature with N values into N binary columns. City=[Mumbai, Delhi, Pune] becomes three 0/1 columns. Use for low cardinality (<20 unique values). Use handle_unknown="ignore" for unseen values at serving time.
Target encodingModel
Replace a category with the mean target value for that category. City → mean house price in that city. Compact and powerful for high cardinality. Must be computed on training data only to avoid leakage.
ImputationModel
Filling missing (null) values with a substitute. Strategies: median (robust to outliers), mean (for normal distributions), most_frequent (for categories), constant (when missing is a signal). Must be fitted on training data only.
Cyclical encodingModel
Encode circular features (hour=0–23, day_of_week=0–6) using sin and cos. Ensures hour 23 and hour 0 are close in feature space. Without this, the model sees them as 23 apart instead of 1 apart.
Lag featureModel
The value of a feature at a previous time step. sales_lag_1d = yesterday's sales. Used in time series to capture autocorrelation. Always use .shift(1) before .rolling() to avoid including the current row.
Rolling averageModel
Average over a sliding time window. 7-day rolling mean of sales captures the weekly trend. Must exclude the current row (shift first) to avoid data leakage.
SHAPModel
SHapley Additive exPlanations. Per-row, per-feature attribution of model predictions. Shows not just which features matter but how and in which direction. The gold standard for ML interpretability.
TF-IDFModel
Term Frequency — Inverse Document Frequency. Weights words by how often they appear in a document (TF) penalised by how common they are across all documents (IDF). Converts text to a sparse numeric matrix.
EmbeddingModel
A dense, low-dimensional vector representation of a high-cardinality entity (word, user, product). Similar entities have nearby embeddings. Sentence transformers produce embeddings for text. Captures semantic similarity.
Feature selectionModel
Choosing which features to include in the model. Filter methods (mutual information), wrapper methods (recursive elimination), embedded methods (L1 regularisation, tree importance). More features ≠ better model.
Model Training
MLflowModel
Open-source platform for managing the ML lifecycle. Tracks experiments (params, metrics, artifacts), stores models in the registry, and serves as the single source of truth for all model versions.
Model registryModel
Centralised store of versioned model artifacts with lifecycle stages: None → Staging → Production → Archived. MLflow provides this. KServe pulls the Production model URI at deploy time.
Hyperparameter tuningModel
Searching for the best model configuration (learning rate, tree depth, number of estimators) that wasn't learned from data. Optuna uses Bayesian optimisation — smarter than grid search.
OverfittingModel
Model performs well on training data but poorly on unseen data. It has memorised the training set instead of learning general patterns. Fix: regularisation, more data, simpler model, cross-validation.
Cross-validationModel
Training and evaluating a model multiple times on different splits of the data. K-fold: split into K parts, train on K-1, test on 1, rotate. Gives a more reliable estimate of true performance than a single split.
Class imbalanceModel
When one class has far fewer samples than another (e.g. 99% non-fraud, 1% fraud). Solutions: SMOTE (oversample minority), class_weight="balanced", threshold tuning on predictions.
Evaluation gateModel
A minimum metric threshold a model must pass before being promoted to production. E.g. test_rmse < 0.50. If it fails, CI exits with code 1 and no deployment happens.
RMSE / MAE / R²Model
Regression metrics. RMSE: root mean squared error — penalises large errors. MAE: mean absolute error — robust to outliers. R²: proportion of variance explained (1.0 = perfect, 0 = no better than mean).
Infrastructure
K3sInfra
Lightweight Kubernetes distribution. Full K8s API with a smaller footprint — ideal for Proxmox home labs. Runs MLflow, KServe, MinIO, Prometheus, Grafana all in one cluster.
KServeInfra
Kubernetes-native model serving platform. Provides auto-scaling, canary rollouts, the v2 inference protocol, transformer/predictor/explainer pods. Loads model artifacts from MinIO at deploy time.
DockerInfra
Containerisation platform. Packages code + dependencies into an image. In MLOps: the serving application runs in Docker. The model itself is NOT baked into the image — it's loaded from MLflow/MinIO at runtime.
FastAPIInfra
Python web framework for building REST APIs. Used as the prediction gateway: handles JWT auth, Pydantic input validation, rate limiting, and request logging before forwarding to KServe.
ProxmoxInfra
Open-source virtualisation platform. Runs your home lab — K3s cluster, MinIO, MLflow all run as VMs or LXC containers on Proxmox. Bare-metal performance with VM isolation.
HelmInfra
Kubernetes package manager. Charts define all K8s resources (Deployment, Service, ConfigMap, etc.) in a template. helm upgrade --install deploys or updates the release with one command.
PostgreSQLInfra
Relational database. Used as the backend store for MLflow tracking server — stores experiment metadata, run params, metrics, tags. Artifact files (models) go to MinIO, not PostgreSQL.
Operations
PrometheusOps
Open-source time-series metrics database. Scrapes /metrics endpoints on pods every 15s. Stores latency histograms, request counts, error rates. AlertManager fires on rule violations.
GrafanaOps
Observability dashboards. Queries Prometheus for infrastructure metrics and MinIO S3 datasource for drift reports. Panels: latency heatmap, prediction distribution, PSI trend, pod health.
EvidentlyOps
Open-source ML observability library. Generates drift reports (HTML + JSON) comparing reference (training) data vs current (production) data. Runs as a K8s CronJob, saves reports to MinIO.
GitHub ActionsOps
CI/CD automation platform. Workflows triggered by git push, PR, or workflow_dispatch (manual/API trigger). Runs lint, tests, dvc repro, Docker build, K3s deploy, smoke test in sequence.
Canary rolloutOps
Deploy a new model version to a small percentage of traffic (e.g. 10%) before promoting to 100%. If metrics degrade, roll back. KServe supports this natively via canaryTrafficPercent.
SLO / SLAOps
Service Level Objective: internal performance target (p99 latency <200ms, error_rate <1%). SLA: contractual commitment to customers. Prometheus AlertManager fires when SLOs are breached.
AlertManagerOps
Prometheus component that handles alert routing. When a rule fires (e.g. p99 >200ms), AlertManager sends notifications to Slack, email, or PagerDuty. Supports silencing and grouping.
Shadow modeOps
Running a new model in parallel with the live model on real traffic without serving its predictions. Compares outputs to catch skew or unexpected behaviour before canary. The safest way to validate a model.
Core Concepts
Train-serve skewConcept
Different feature transforms at training time vs serving time causing silent prediction errors. The #1 cause of production ML failures. Prevented by serialising the entire sklearn Pipeline (imputer + scaler + model) as one artifact.
Data driftConcept
Statistical shift in input feature distributions between training time and serving time. Example: training MedInc mean was 3.87, serving requests have mean 2.10. Model predictions become systematically biased.
Concept driftConcept
The relationship between features and target changes. Same inputs, different correct outputs. Example: fraud patterns evolve — old fraud patterns become normal, new ones emerge. Requires ground-truth labels to detect.
Data leakageConcept
Future or target information bleeding into training data. Causes artificially high accuracy in notebooks that collapses in production. Types: target leakage, temporal leakage, scaler fitted on full dataset before split.
ReproducibilityConcept
Given a git SHA, being able to recreate the exact dataset (via DVC), features, model, and evaluation results from any past experiment. The core promise of MLOps. MLflow + DVC together provide this.
PSI (Population Stability Index)Concept
Measures how much a distribution has shifted. PSI <0.1 = stable (no action). 0.1–0.2 = monitor closely. >0.2 = significant drift — alert and consider retraining. Used by Evidently and production banks worldwide.
KS testConcept
Kolmogorov-Smirnov test. Non-parametric statistical test comparing two continuous distributions. p-value <0.05 means the distributions are significantly different — drift detected.
Inference latency (p99)Concept
The 99th percentile response time for prediction requests. The slowest 1% of requests. Production SLO target is typically p99 < 200ms. Monitored by Prometheus, visualised as a heatmap in Grafana.
Cold startConcept
The first inference request after a model pod starts from zero replicas. The pod loads the model from MinIO (2–10 seconds) before responding. Mitigate with min_replicas=1 for latency-sensitive endpoints.
Model cardConcept
Documentation describing a model's intended use, performance across different subgroups, training data, known limitations, and ethical considerations. Required for governance and regulatory compliance.
Online vs batch inferenceConcept
Online: single real-time request → immediate response (<200ms). Batch: entire dataset processed asynchronously (no latency constraint). Choose based on use case — fraud detection = online, monthly churn scoring = batch.
A/B testingConcept
Running two model versions simultaneously on production traffic to compare real-world business impact (revenue, clicks, errors). More reliable than offline metrics alone. Typically run for 48–72 hours for statistical significance.
MLOpsConcept
Machine Learning Operations. Applying software engineering (DevOps, CI/CD, testing, monitoring) to the ML lifecycle. Goal: making model deployment reliable, repeatable, and continuously improving. The entire field this site documents.
Feedback loopConcept
Using production predictions and their outcomes to improve the next model version. User corrections, downstream system signals, explicit ratings — all become training data. Closes the loop between deployment and improvement.
ONNXConcept
Open Neural Network Exchange. A portable model format enabling models trained in PyTorch or TensorFlow to be served in other runtimes (C++, Java, ONNX Runtime). Also enables int8 quantisation for faster inference.
QuantisationConcept
Reducing model precision from float32 to int8 (or int4). Makes models 4x smaller and 2–4x faster at inference time with minimal accuracy loss. Used when memory or latency is a constraint.
