Data sources — where it all begins

Every ML system is only as good as the data feeding it. Sources define the raw material: its type, freshness, reliability, and legal constraints. Understanding your source is the first thing you do before writing a single pipeline.

Structured / DB
Relational tables, data warehouses (PostgreSQL, BigQuery, Redshift). High quality, schema-enforced. Must be snapshotted via ETL — never read live in training.
ETL requiredhigh structure
Streaming / event
Kafka topics, webhooks, IoT sensors, clickstreams. High volume, low latency. Challenging to reproduce. Must land in MinIO with event-time partitioning.
append-onlytime-ordered
Files / uploads
CSV, JSON, Parquet, images, PDFs, audio. From S3, SFTP, local disk, or user upload. Variable schema. AutoManager handles upload routing to MinIO.
schema variesmanual ingestion
External APIs
Weather, finance, social, third-party enrichment. Rate-limited, paid, unreliable. Always cache in MinIO with TTL — never call live during training.
rate-limitedcache required
Synthetic / augmented
Generated data for rare classes, privacy compliance (GDPR anonymisation), data augmentation (image flips, noise). Must be clearly labelled and versioned separately.
label carefullyversioned separately
Feedback / labels
Production predictions labelled by humans or implicit signals (clicks, corrections). The most valuable data. Closes the training loop. Treated as a first-class source.
closes loopgold standard
Common mistake: reading from a live database during training. A row updated mid-run makes your dataset non-reproducible. Always ETL-snapshot to parquet first.
Your home lab rule: all sources must land in MinIO first — no exceptions. This is the architectural gate that makes everything downstream reproducible.

Ingestion — getting data into the platform

Ingestion is the controlled act of moving raw data from source systems into your ML platform's landing zone. The golden rule: MinIO is the first and only destination. Nothing is processed until it has landed and been acknowledged.

Batch ingestion pattern
Scheduled job (cron or Airflow DAG) runs ETL: extract from DB → transform to parquet → load to minio://mlops-raw/source=db/date=YYYY-MM-DD/. Date-partitioned for incremental loads. DVC then snapshots the new partition.
Stream ingestion pattern
Kafka consumer reads events → micro-batches every N minutes → writes to minio://mlops-raw/source=events/dt=YYYY-MM-DD/hr=HH/. Event-time partitioning (not processing-time) ensures reproducibility.

Data validation — trust but verify

Raw data is never clean. Validation is the quality gate between raw ingestion and processed features. It catches problems early — before they silently corrupt a model or cause an outage 3 months later.

Schema validation
Does the data match the expected types, columns, and nullable constraints? Tools: Great Expectations, Pandera, or Pydantic. Run on every new partition before processing.
PanderaGreat Expectations
Statistical checks
Are distributions reasonable? Check: mean ± 3σ for numerics, null rate < threshold, cardinality for categoricals. Flag outliers but don't silently drop them.
Z-scoreIQR bounds
Business rule checks
Domain-specific: age > 0, price > 0, date not in future, foreign keys resolve. These are the ones you only discover when a model makes a crazy prediction.
domain logicsanity bounds
Validation outcome actions
PassProceed to feature engineering. Log metrics.
WarnProceed but alert. Log anomaly for review.
Fail (recoverable)Quarantine batch. Retry after fix. Do not block pipeline.
Fail (critical)Halt pipeline. Page on-call. Write incident report to MinIO.
Pandera quick example
import pandera as pa schema = pa.DataFrameSchema({ "age": pa.Column(int, pa.Check.ge(0)), "price": pa.Column(float, pa.Check.gt(0)), "label": pa.Column(str, pa.Check.isin( ["fraud","legit"] )), }, nullable_checks=False) schema.validate(df) # raises on fail
Never silently drop failed rows. Log every failure to MinIO with the row index and reason. Silent drops create invisible bias — your model learns from a filtered distribution, not the real one.

Data versioning — reproducibility is everything

A model is not just weights — it's weights + the exact dataset it was trained on. DVC is the Git for data: it tracks what changed, stores it in MinIO, and lets you reproduce any past experiment perfectly.

How DVC works
DVC replaces large files in Git with tiny .dvc pointer files (MD5 hash + path). The actual data stays in MinIO. git checkout + dvc pull reconstructs any historical state.
DVC pipeline stages
dvc.yaml defines DAG: ingest → validate → features → split → train. Each stage has inputs, outputs, params. dvc repro only re-runs what changed (content-addressed caching).
MLflow integration
At training time, log the DVC commit hash as an MLflow run param. Now every model knows its exact data lineage back to the raw source.

Feature engineering — raw data → model fuel

Features are the language your model speaks. Raw data is rarely in that language. Feature engineering is the translation layer — and it's often where 80% of the model's predictive power comes from, not the algorithm itself.

Numeric transformations
Normalisation (min-max), standardisation (z-score), log transform for skewed distributions, binning continuous variables, polynomial features for interactions.
StandardScalerPowerTransformer
Categorical encoding
One-hot (low cardinality), ordinal (ordered categories), target encoding (high cardinality), embedding lookup (very high cardinality / text). Handle unseen categories at serve time.
OHETarget encEmbeddings
Temporal features
Day-of-week, hour-of-day, is_weekend, days_since_event, rolling averages (7d/30d), lag features. Always computed from event_time, never ingest_time.
rolling avglag features
Training vs serving skew — the silent killer
The #1 cause of model degradation in production. Happens when the feature computation at training time differs from serving time. Example: training uses pandas mean imputation on the full dataset; serving computes mean on the last 100 rows. The values differ.
Fix: use a sklearn Pipeline that is serialised with the model. The same code path must run at train and serve time.
Feature store concept
Centralised registry of feature definitions, computed values, and serving endpoints. Online store (Redis) for low-latency serving; offline store (MinIO/Parquet) for training.
Your setup: use sklearn Pipelines serialised into the MLflow artifact. MinIO acts as the offline feature store. When you hit scale, introduce Feast or Hopsworks.
FeastHopsworkssklearn Pipeline
Pipeline that prevents train/serve skew
from sklearn.pipeline import Pipeline from sklearn.preprocessing import StandardScaler from sklearn.impute import SimpleImputer from sklearn.ensemble import RandomForestRegressor pipe = Pipeline([ ("impute", SimpleImputer(strategy="median")), ("scale", StandardScaler()), ("model", RandomForestRegressor(n_estimators=100)), ]) pipe.fit(X_train, y_train) import mlflow mlflow.sklearn.log_model(pipe, "model") # The WHOLE pipeline (imputer + scaler + model) is saved. # At serve time: pipe.predict(X_new) — same transforms, guaranteed.

Data splitting — the three-set discipline

How you split data is a direct proxy for how honest your model evaluation is. Get it wrong and your metrics look great in notebooks but fail in production. The three-set pattern maps perfectly to SRE's dev/staging/prod thinking.

Training set (~70%)
The only data the model ever "sees" during fitting. Used for gradient updates, weight adjustments, and learning patterns. Think: development environment. Iterate freely.
model sees this~70% of data
Validation set (~15%)
Used to tune hyperparameters and select the best model. Your staging environment — test before committing. The model never trains on this. But you look at it, so it has indirect influence.
for tuning~15% of data
Test set (~15%)
Locked away. Touched exactly once — final evaluation before production release. Your production gate. If you look at it more than once, it's contaminated and becomes val.
one-time gate~15% of data
Splitting strategies
Random splitFor i.i.d. data. sklearn train_test_split with fixed random_state.
Stratified splitFor imbalanced classes. Preserves class ratio in all splits.
Temporal splitFor time-series. Train on past, val/test on future. Never shuffle.
Group splitWhen rows belong to groups (per-user). Keep all rows of a group together.
Data leakage — the silent error
Target leakageFeature derived from the target variable
Future leakageUsing t+1 data to predict t
Scaler leakFitting scaler on full dataset before split
Duplicate rowsSame row appearing in both train and test
Correct stratified split with index persistence
from sklearn.model_selection import train_test_split import numpy as np X_train, X_tmp, y_train, y_tmp = train_test_split( X, y, test_size=0.30, random_state=42, stratify=y) X_val, X_test, y_val, y_test = train_test_split( X_tmp, y_tmp, test_size=0.50, random_state=42, stratify=y_tmp) # Save exact indices to DVC for reproducibility np.save("data/splits/train_idx.npy", X_train.index) np.save("data/splits/val_idx.npy", X_val.index) np.save("data/splits/test_idx.npy", X_test.index) # dvc add data/splits/ — version the exact split
Always fit preprocessing (scalers, encoders, imputers) on training data only, then transform val and test. Fitting on the full dataset before splitting is the most common leakage mistake.

Data in training — how the model consumes data

Training is not just "pass data to model.fit()". At scale, how data is loaded, batched, shuffled, and augmented determines speed, convergence, and the quality of what the model learns.

Batch loading
Data loaded in mini-batches (e.g. 32–256 rows). Never load the full dataset into RAM. Use pandas read_parquet with column selection, or PyTorch DataLoader with lazy reading from MinIO.
Shuffling
Shuffle training data each epoch to break order bias. Do NOT shuffle validation or test sets. For time-series: do not shuffle at all — temporal order is the signal.
Data augmentation
Creating synthetic variants: image flips/crops/noise, text back-translation, SMOTE for class imbalance, Gaussian noise on tabular numerics. Applied only to training data — never val/test.
Class imbalance handling
OversamplingSMOTE — generate synthetic minority samples. Applied after train split.
UndersamplingRandomly drop majority class rows. Risk: losing signal.
Class weightssklearn class_weight="balanced" — scale loss by inverse frequency.
Threshold tuningMove decision boundary post-training. Cheapest option.

Serving data — data at inference time

When a model runs in production, data arrives one row at a time, in real time, from an external caller. The challenges here are latency, consistency with training, and handling inputs you never saw during training.

Online features
Features computed at request time from live data. Must be fast (<10ms). Served from Redis or a low-latency KV store. Examples: user's last 5 clicks, current session length.
Precomputed features
Features computed in batch and cached. Looked up by entity key (user_id, product_id) at request time. Slightly stale (minutes to hours) but very fast. Stored in Redis or DynamoDB.
Request payload
Raw data sent by the caller — validated by Pydantic schema, then transformed through the same sklearn Pipeline that was trained. The pipeline handles all feature engineering identically to training.
Handling unseen values at serving
Unknown categoryMap to "unknown" token — must be in training vocab. Never return 500.
Missing featuresImpute with training median/mode (stored in Pipeline). Never forward null to model.
Out-of-range numericClip to [min, max] seen in training. Log the clip for drift monitoring.
Schema changeReturn 422 with clear error. Do not guess at missing columns.
Request + response logging to MinIO
{ "request_id": "uuid-v4", "ts": "2025-08-01T14:32:00Z", "model_version": "rfr-v3", "input_features": { "MedInc": 8.3252, "HouseAge": 41.0, "AveRooms": 6.98 }, "prediction": 412500.0, "confidence": 0.87, "latency_ms": 23 }
Training-serving skew: if your FastAPI endpoint applies any transformation that differs from the sklearn Pipeline inside the model artifact, predictions will be biased. The pipeline object must be the single source of truth for all transforms.

Data drift & monitoring — detecting when data changes

Models degrade because the world changes, not because the code breaks. Drift monitoring is the early warning system: watching the statistical distribution of incoming data and comparing it against what the model was trained on.

Data drift (input)
The distribution of input features shifts. Example: a new city is added, changing the geographic distribution. The model hasn't seen these patterns. Performance degrades invisibly.
PSI testKS test
Concept drift (label)
The relationship between features and target changes. Example: fraud patterns evolve — old patterns stop being fraudulent, new ones emerge. Model accuracy drops even if inputs look the same.
hardest to detectneeds labels
Prediction drift
The distribution of model outputs shifts. Easier to detect than concept drift because you don't need labels. A proxy for both data drift and concept drift. Monitor this first.
no labels neededearly signal
Statistical tests used
PSIPopulation Stability Index. <0.1 stable, 0.1–0.2 watch, >0.2 alert.
KS testKolmogorov-Smirnov for continuous distributions. p < 0.05 = drift.
Chi-squaredFor categorical features. Compares frequency tables.
Jensen-ShannonSymmetric KL divergence. Works on any distribution.

Data lineage & governance — the full audit trail

Lineage is the ability to answer: where did this data come from, what transformations were applied, who touched it, and which models were trained on it? Governance adds the policy layer: who can access what, under what conditions.

Lineage chain
Source system → ingest job → MinIO raw → DVC snapshot → feature pipeline → MLflow experiment → model version → KServe deployment → inference logs. Every link is traceable.
How MLflow + DVC provide it
MLflow run stores: DVC hash (→ exact dataset), git SHA (→ exact code), params.yaml (→ exact config). Given an MLflow run_id, you can reconstruct every artifact and the data that produced it.
Governance basics
Who can read raw PII data? Who can promote models to production? MinIO bucket policies + IAM roles enforce access. Audit logs track all S3 API calls for compliance (GDPR, SOC2).
Lineage query: "which model used this data?"
import mlflow client = mlflow.tracking.MlflowClient() runs = client.search_runs( experiment_ids=["1"], filter_string=( "params.dvc_data_hash = 'abc123def456'" ) ) for r in runs: print( r.info.run_id, r.data.tags.get("mlflow.runName") )
