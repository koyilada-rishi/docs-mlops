Stage 01

Data Pipeline — Sources to MinIO to DVC

Every ML system lives or dies on its data. This stage covers how data moves from every source type into MinIO as the mandatory first landing zone, how it gets validated, and how DVC takes over for reproducible versioning.

The golden rule
All data from any source must land in MinIO first — before any processing, before any model sees it, before any feature is computed.

This is the architectural gate that makes every downstream stage reproducible. If a pipeline reads from a live database, a different result comes back each run. If it reads from MinIO, the same bytes are there every time.
Data source types
Structured / Database
PostgreSQL, MySQL, BigQuery, Redshift, Snowflake. Schema-enforced, high quality. Must be ETL-snapshotted to parquet — never read live during training. Partitioned by date.
ETL requiredhigh structureschema enforced
Streaming / Events
Kafka topics, webhooks, IoT sensors, clickstreams. High volume, low latency. Must land in MinIO with event-time partitioning (not ingest-time — they differ by minutes to hours).
append-onlytime-orderedmicro-batch
Files / Uploads
CSV, JSON, Parquet, images, PDFs, audio. From S3, SFTP, local disk, or user upload. Variable schema. AutoManager handles routing to MinIO. Schema validated on arrival.
schema variesmanual ingestion
External APIs
Weather, finance, social, third-party enrichment. Rate-limited, paid, unreliable. Always cache in MinIO with TTL — never call live during training. Snapshot once per day max.
rate-limitedcache required
Synthetic / Augmented
SMOTE for class imbalance, image augmentation (flips/crops/noise), GDPR anonymisation. Must be clearly labelled as synthetic and versioned separately from real data.
label carefullyseparate version
Feedback / Labels
Production predictions labelled by humans or implicit signals (clicks, corrections, returns). The most valuable data. Closes the training loop. Treated as a first-class source in MinIO.
closes loopgold standard

Two MinIO locations — raw zone vs DVC cache
Your data lives in TWO MinIO locations with completely different purposes. This is intentional architecture, not duplication waste.
mlops-raw/ — Raw landing zone
Managed by: Ingestion pipeline / AutoManager
Written by: ETL jobs, Kafka consumers, file uploads
Path format: Human-readable — source=db/date=2025-08-01/
Lifecycle: Expire after 90 days — once DVC has tracked it, this copy is expendable
Purpose: Immutable audit record of exactly what the source system delivered
mc ilm add myminio/mlops-raw --expiry-days 90 # MinIO auto-deletes raw/ objects after 90 days # The DVC cache copy is the durable one
mlops-dvc-cache/ — DVC remote
Managed by: DVC (dvc push / dvc pull)
Written by: dvc push command only
Path format: Content-addressed — files/md5/ab/c123def456.../
Lifecycle: Permanent. Never deleted unless dvc gc is run.
Purpose: Versioned, reproducible, deduplicated working copy
Deduplication: If two files have the same content, DVC stores only one copy regardless of how many .dvc files point to it. Same content = same MD5 = one entry.
Data validation — two layers
Layer 1: Raw validation (after ingestion, before DVC)
Runs on every new partition. Asks: is this data sane?

Schema check — correct columns, types, nullable constraints
Statistical check — mean ± 3σ bounds, null rate < threshold
Business rules — age > 0, price > 0, date not in future
Volume check — row count within ±20% of historical average
import pandera as pa schema = pa.DataFrameSchema({ "age": pa.Column(int, pa.Check.ge(0)), "price": pa.Column(float, pa.Check.gt(0)), "label": pa.Column(str, pa.Check.isin(["fraud","legit"])), }) schema.validate(df) # raises on fail — halts pipeline
Layer 2: Model-specific validation (after feature engineering)
Runs before training starts. Asks: is this data ready for THIS specific model?

Feature presence — all expected columns exist in correct order
Distribution check — within range of what model was designed for
Null-free check — after imputation, no nulls remain
Leakage check — target column not in feature set
These are two different validations done by different tools for different reasons. Raw validation is pipeline-level quality control. Model-specific validation is model-level contract checking.

First-time bootstrap — how to register new data with DVC
import boto3, pathlib # Step 1: download from raw landing zone to local disk # (NOT dvc pull — no pointer exists yet) s3 = boto3.client("s3", endpoint_url="http://192.168.1.15:30900", aws_access_key_id="minioadmin", aws_secret_access_key="minioadmin") pathlib.Path("data/raw").mkdir(parents=True, exist_ok=True) s3.download_file("mlops-raw", "source=db/date=2025-08-01/train.parquet", "data/raw/train.parquet") # Step 2: tell DVC to track it (creates .dvc pointer file) $ dvc add data/raw/train.parquet # creates: data/raw/train.parquet.dvc (MD5 hash pointer) # adds to: data/raw/.gitignore (actual file NOT in git) # Step 3: upload to DVC cache in MinIO $ dvc push # → minio://mlops-dvc-cache/files/md5/ab/c123.../ # Yes — same data now in two MinIO locations. This is fine. # Step 4: commit the pointer to Git $ git add data/raw/train.parquet.dvc data/raw/.gitignore $ git commit -m "data: track training dataset 2025-08-01" $ git push

Ingestion patterns
Batch ingestion (database / file sources)

Stream ingestion (Kafka → MinIO)
from kafka import KafkaConsumer import pandas as pd, boto3, json from datetime import datetime consumer = KafkaConsumer( "user-events", bootstrap_servers=["kafka:9092"], auto_offset_reset="earliest") batch, BATCH_SIZE = [], 1000 for msg in consumer: event = json.loads(msg.value) batch.append(event) if len(batch) >= BATCH_SIZE: df = pd.DataFrame(batch) # Partition by EVENT TIME — not ingest time! event_dt = pd.to_datetime(df["event_ts"]).dt key = (f"source=events/" f"dt={event_dt.date.min()}/" f"hr={event_dt.hour.min()}/batch.parquet") s3.put_object(Bucket="mlops-raw", Key=key, Body=df.to_parquet(index=False)) batch = []
Common data pipeline failures
Partial writesNetwork drops mid-upload → corrupt file in MinIO. Fix: write to a temp key, then rename atomically after checksum verification.
Schema driftSource DB adds/renames a column → downstream feature code breaks. Fix: schema hash check at ingest — alert if hash changes.
Duplicate ingestionETL job re-runs, data doubles in MinIO. Fix: idempotent write key based on source + date + checksum. Dedup at read time.
Timezone mismatchServer UTC vs app local time → lag features off by hours. Fix: store all timestamps as UTC. Convert at last mile display only.
Event time vs ingest time confusionA sale at 11pm ingested at 2am. Using ingest time for day_of_week gives wrong signal. Fix: always use event_ts, not ingest_ts.
Live DB read during trainingThe row you read at 9am is different from the same row read at 2pm. Fix: ETL snapshot to parquet before training. Never read live.
Missing DVC pushEngineer does dvc add but forgets dvc push. Teammate runs dvc pull and gets nothing. Fix: add dvc push to CI pipeline — it's a no-op if already pushed.
