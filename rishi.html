Key terminology

The 20 most important MLOps terms. See full 100+ term glossary →

MLOps
Machine Learning Operations — applying DevOps principles to the ML lifecycle to make model deployment reliable and repeatable.
Feature store
Centralised system that stores, serves, and manages ML features. Online store (Redis) for serving, offline (MinIO) for training.
Data drift
Statistical shift in input feature distributions between training time and serving time. Detected by PSI or KS test.
Concept drift
The relationship between features and the target changes. Harder to detect — requires ground-truth labels from production.
Train-serve skew
Different feature transforms at training vs serving time causing silent prediction errors. Prevented by serialising the full sklearn Pipeline.
Model registry
Centralised store of versioned model artifacts with lifecycle stages: None → Staging → Production. MLflow provides this.
KServe
Kubernetes-native model serving platform. Provides auto-scaling, canary rollouts, and the v2 inference protocol on top of K3s.
DVC
Data Version Control. Git for data — tracks dataset versions via MD5 hash pointers stored in Git, actual bytes in MinIO.
Canary rollout
Sending a small % of traffic (e.g. 10%) to a new model version before promoting it to 100%. Risk-free deployment pattern.
PSI
Population Stability Index. Measures distribution shift. <0.1 stable, 0.1–0.2 watch, >0.2 alert and retrain.
Inference latency
Time from receiving a prediction request to returning a response. SLO target is usually p99 < 200ms.
Data lineage
The ability to trace any model back to the exact dataset, code, and config that produced it. MLflow run + DVC hash provides this.
Evaluation gate
A metric threshold a model must pass before being promoted to production. E.g. RMSE < 0.5 on the held-out test set.
Reproducibility
Given a git SHA, being able to recreate the exact dataset, features, model, and evaluation results from any past experiment.
A/B testing
Running two model versions simultaneously on production traffic to compare real-world performance, not just offline metrics.
Feature importance
A measure of how much each feature contributes to model predictions. SHAP gives per-prediction, per-feature attribution.
Hyperparameter
Configuration that controls training (learning rate, tree depth, number of estimators) — not learned from data, set before training.
Shadow mode
Running a new model in parallel with the live model without serving its predictions. Compares outputs to catch skew before rollout.
Cold start
The first inference request after a model pod starts. May be slow due to model loading from MinIO. Mitigated by pre-warming.
ONNX
Open Neural Network Exchange — a portable model format that allows models trained in one framework to be served in another.
