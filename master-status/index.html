Quality Gate Script

# quality_gate.py
import mlflow, sys, os

client = mlflow.tracking.MlflowClient(os.environ["MLFLOW_TRACKING_URI"])
runs = client.search_runs(
    experiment_ids=["1"],
    order_by=["start_time DESC"],
    max_results=1
)

r2   = float(runs[0].data.metrics["r2"])
rmse = float(runs[0].data.metrics["rmse"])

print(f"R2: {r2:.4f} | RMSE: {rmse:.4f}")

if r2 < 0.80:
    print(f"FAILED: R2 {r2:.4f} below 0.80")
    sys.exit(1)

if rmse > 0.60:
    print(f"FAILED: RMSE {rmse:.4f} above 0.60")
    sys.exit(1)

print("PASSED: Model meets quality thresholds")

SECTION 5 · QUESTIONSPipeline Questions

Questions about your specific pipeline — know every detail.

"Walk me through your complete MLOps pipeline from training to monitoring."
STRONG ANSWER
"Training runs on a dedicated Windows GPU node registered as a self-hosted GitHub Actions runner. When training code is pushed, the workflow triggers, trains a RandomForest model on California Housing data, and logs everything to MLflow — parameters, RMSE, R2, and the model artifact in MinIO. A quality gate checks metrics before deployment proceeds. If R2 is below 0.8, the pipeline fails. If it passes, the workflow applies the updated KServe InferenceService manifest. KServe pulls the model from MinIO via MLflow registry and starts serving. Every prediction request goes through our custom FastAPI predictor — prediction is returned immediately while a background task logs input features and output to MinIO as parquet. An hourly K3s CronJob runs Evidently, compares production prediction logs against training reference data, generates drift reports visible in Evidently UI, and pushes summary metrics to Prometheus Pushgateway. Grafana shows both infra metrics and drift metrics in unified dashboards."
"How do you handle model versioning across training, registry, and serving?"
STRONG ANSWER
"MLflow Model Registry is the single source of truth. Every training run logs to MLflow and registers a new version. Versions have stages — None, Staging, Production, Archived. The KServe InferenceService references the registry by name and version: models:/best_random_forest_model/1. When we promote a new version to Production in MLflow, we update the InferenceService manifest to point to the new version and apply it. KServe handles the pod rollover gracefully. The old version stays in the registry as Archived for rollback reference."
"Why did you use a custom FastAPI predictor instead of KServe's built-in sklearn runtime?"
STRONG ANSWER
"The built-in runtime is a black box — you can't add logging, custom validation, or business logic. For production monitoring we need to log every prediction asynchronously to MinIO. The custom predictor gives us full control: we own the model loading, input validation, prediction logic, and logging. The trade-off is we maintain more code, but the observability we gain is essential for drift detection. In production at scale, we'd use the KServe Transformer pattern — keep the built-in runtime and add a transformer for pre/post processing and logging. That's a cleaner separation of concerns."
"How does your prediction logging work without affecting latency?"
STRONG ANSWER
"FastAPI's BackgroundTasks runs after the response is sent to the client. So the user gets their prediction in 5ms while the logging happens in the background without them waiting. We also batch — instead of writing to MinIO on every request, we accumulate 50 predictions in an in-memory buffer protected by a threading lock for concurrency safety. When the buffer hits 50, we write one parquet file. This reduces MinIO PUT operations by 50x. On pod shutdown, force_flush() saves whatever is in the buffer so no data is lost on graceful shutdown."

SECTION 5 · QUESTIONSML Knowledge Questions

"What's the difference between data drift and concept drift?"
STRONG ANSWER
"Data drift means the input feature distributions change — P(X) changes but the relationship between features and outcome P(Y|X) stays the same. For example, if you trained on houses with average age 20 years but production data now averages 35 years, that's data drift. The model might still work if it generalizes well. Concept drift is more fundamental — P(Y|X) changes, meaning the relationship between inputs and outputs has shifted. Same features now produce different correct outputs. This happens when the world changes — like housing market dynamics shifting post-pandemic. Concept drift is harder to detect because input features can look normal but model performance degrades. Data drift you catch with PSI and KS tests. Concept drift you catch with performance monitoring against actuals."
"How do you handle class imbalance in production models?"
STRONG ANSWER
"At training time — SMOTE oversampling, class weights, or threshold optimization using the validation set precision-recall curve. Don't use accuracy as your metric — use F1 or AUC-ROC. At monitoring time — imbalance matters for alert thresholds. If you expect 1% fraud rate and predictions suddenly show 5% positives, that's an alert. Monitor the positive prediction rate as a separate metric in Prometheus. Also use PR-AUC instead of ROC-AUC for monitoring because PR-AUC is more sensitive to minority class performance."
"What is PSI and how do you interpret it?"
STRONG ANSWER
"PSI is Population Stability Index — it measures how much a feature's distribution has shifted between training and production. It bins the feature into buckets, calculates the percentage of data in each bucket for training vs production, and computes a weighted log ratio. Interpretation: below 0.1 means stable, 0.1 to 0.2 means slight drift worth monitoring, above 0.2 means significant drift that likely impacts model performance, above 0.25 means major drift requiring immediate investigation and likely retraining. It's especially common in financial ML — credit scoring teams use it extensively."

SECTION 5 · QUESTIONSSystem Design Questions

"Design an ML platform that can handle 1 million predictions per day."
STRONG ANSWER — STRUCTURE YOUR ANSWER LIKE THIS
"First I'd clarify: single model or multiple? Real-time or batch? Latency SLA? For 1M/day at 50 req/sec peak with 100ms p99 SLA: Serving: KServe with minReplicas=3, maxReplicas=20, HPA on request rate. Multiple nodes in K8s cluster. "How would you implement A/B testing for ML models?"
STRONG ANSWER
"A/B testing in ML is different from web A/B testing — you're comparing business outcomes, not just technical metrics. Technically in KServe: two InferenceServices running simultaneously with traffic split, a routing layer sending users to the same model consistently using user ID hashing. Log model version with every prediction. When actuals come back, join predictions with outcomes by model version. Calculate business metrics (conversion rate, revenue) per model version. Run statistical significance test (t-test) on the difference. If significant and model B wins, promote it. The key is consistent user assignment — same user always hits same model to avoid confounding effects."

SECTION 5 · QUESTIONSDebugging Questions

"Your model is working in staging but performing poorly in production. How do you debug?"
STRONG ANSWER
"This is almost always training-serving skew or environment differences. My investigation order: First, compare feature distributions — log the actual input features both environments receive and compare histograms. If distributions differ, the data pipeline is inconsistent. Second, check if feature engineering code is truly identical between environments. Third, check environment differences — library versions, Python versions, available memory. A model that barely fits in staging RAM might OOM in production under concurrent load. Fourth, check the prediction request format — staging tests often use simplified inputs. Fifth, run a production request through staging with the exact same payload. If staging then fails, you've isolated the input as the cause."
"RMSE suddenly spiked in production. Walk me through your debugging process."
STRONG ANSWER
"First: is it infra or ML? Check if latency also spiked — if yes, it's likely infra. Check pod resources, network, KServe logs. If latency is fine, it's ML. Second: was there a recent deployment? Check git log. If yes, rollback immediately via canaryTrafficPercent=0, then investigate offline. Third: check Evidently drift report — did PSI spike for any feature? If yes, input data distribution changed, model is working correctly on wrong distribution, need to retrain. Fourth: check for training-serving skew — log actual feature values model receives and compare against training data statistics. Fifth: check for concept drift — features look normal, no deployment, but RMSE still degraded. The world changed. Retrain with recent data."

SECTION 5 · QUESTIONSScaling Questions

"How would you manage 100 models in production?"
STRONG ANSWER
"At 100 models, manual management breaks down. Three key things: First, templated pipelines — one GitHub Actions workflow template with model_name as parameter. Any model triggers the same pipeline with different config. Second, MLflow Registry as source of truth — a deployment loop reads all models tagged Production and ensures they're deployed in KServe. No manual InferenceService updates. Third, KServe ModelMesh instead of one pod per model — ModelMesh runs multiple models on shared infrastructure, loading and unloading models based on traffic. At 100 models you also need a model catalog — what's the model's purpose, owner, SLA, data source, retrain schedule. Without that, 100 models becomes unmaintainable."
"When would you add Kafka to your prediction logging pipeline?"
STRONG ANSWER
"When direct MinIO writes become a bottleneck — roughly above 500-1000 requests per minute. Kafka absorbs traffic spikes — if MinIO has a 10-second slowdown, requests pile up in Kafka rather than backing up to the predictor. Kafka also gives exactly-once semantics — no duplicate logs even if the consumer restarts. And Kafka enables multiple consumers — Evidently reads from the same topic, as does a real-time dashboard, as does a data warehouse. Without Kafka you'd have to write to MinIO multiple times for multiple consumers. The trade-off is operational complexity — Kafka needs its own cluster, monitoring, and maintenance. For our current scale, direct MinIO batching is fine."

SECTION 5 · QUESTIONSGPU Questions

"How do you manage GPU resources for ML training in Kubernetes?"
STRONG ANSWER
"GPU scheduling in K8s requires the nvidia device plugin — a DaemonSet that runs on every GPU node and exposes GPU capacity to the K8s scheduler. Training jobs request GPU via resource limits: nvidia.com/gpu: 1. K8s schedules the job only on nodes with available GPU. For efficient GPU utilization, time-slicing allows multiple jobs to share one GPU — useful when jobs don't need full GPU memory. For serving, KServe InferenceService requests GPU in the same way. For large model serving, Triton Inference Server handles multiple models on one GPU with dynamic batching. On our home lab, the GTX 1650 isn't connected to the K3s cluster yet — sklearn models don't need GPU. For deep learning serving we'd register the Windows node as a K8s GPU worker node."
"Your team wants to serve a large language model. How do you approach this?"
STRONG ANSWER
"Several layers to consider. Infrastructure: LLM serving needs high VRAM — A100 80GB or H100 for large models. On-prem this is expensive so cloud GPU instances are more practical. Serving framework: vLLM for throughput-optimized inference, Triton for multi-model serving, or specialized solutions like TGI. KServe supports custom containers so vLLM can run behind KServe for standard routing and autoscaling. Optimization: quantization (INT8, INT4) reduces memory by 4x with minimal quality loss. Batching: dynamic batching in vLLM groups requests together for GPU efficiency. Monitoring: LLM-specific metrics — token throughput, time-to-first-token latency, KV cache hit rate. Drift detection for LLMs is different — embedding drift rather than feature drift."

SECTION 6 · SECURITYS3 / MinIO Security

How to secure object storage access in private clusters — production patterns.

The Problem With Your Current Setup

Current (wrong for production):
  minioadmin / minioadmin — root credentials everywhere
  Same credentials for predictor, evidently, mlflow
  Credentials in plain YAML files
  No bucket access restrictions
  Anyone with credentials can access everything

Production Pattern — Least Privilege

Create separate MinIO service accounts:
  mlflow-sa:
    → read/write mlflow-artifacts/* only
  
  predictor-sa:
    → write predictions/* only
    → nothing else
  
  evidently-sa:
    → read predictions/*
    → read dvc-data/*
    → write evidently/*
    → nothing else

Root credentials → admin only, never in application code

AWS Production Pattern — IRSA

IAM Roles for Service Accounts (IRSA):
  K8s ServiceAccount → linked to IAM Role
  Pod assumes role automatically via token projection
  No access keys stored anywhere
  Credentials rotate automatically
  If pod is compromised → limited blast radius

Your equivalent:
  kserve-sa ServiceAccount → MinIO policy (predictor-sa)
  cronjob ServiceAccount → MinIO policy (evidently-sa)

VPC Endpoint for S3

In AWS:
  VPC Endpoint for S3 → traffic never leaves AWS network
  No internet exposure
  Much faster than public S3 access
  Cheaper — no data transfer costs

Your equivalent:
  Pod uses: http://minio.mlflow.svc.cluster.local:9000
  Traffic stays inside K8s cluster network
  Never goes through public internet ✅ (you already do this)

Interview Answer

Say This
"In production we'd use IAM Roles for Service Accounts — pods assume IAM roles without storing credentials anywhere. Bucket policies enforce least privilege — the predictor can only write to predictions bucket, Evidently can only read from it. In our home lab we use K8s Secrets with MinIO service accounts scoped per component. In transit traffic uses cluster-internal DNS so it never leaves the private network. At rest, MinIO supports AES-256 encryption for sensitive buckets."

Lifecycle Policy Per Bucket

BucketPolicyReason

mlflow-artifactsKeep foreverModels are precious, small size
dvc-dataKeep foreverTraining data audit trail
evidentlyDelete after 180 daysReports are historical reference only

Set In MinIO

# Via MinIO Console:
# Bucket → Lifecycle → Add Rule → Expiry: 90 days

# Via mc CLI:
mc ilm add local/predictions --expiry-days 90
mc ilm add local/evidently --expiry-days 180

S3 Storage Tiers (AWS)

Recent predictions (0-30 days):   S3 Standard
Older predictions (30-90 days):   S3 Infrequent Access (cheaper)
Archived reports (90+ days):      S3 Glacier (very cheap)
Delete after:                     90 days for predictions

This reduces storage cost by 70% vs keeping everything in Standard.
