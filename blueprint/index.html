Raw Data / DVC
Data versioning — foundation of reproducibility

What It Is

DVC (Data Version Control) treats data like Git treats code. Every dataset version is tracked with a SHA-256 hash. The actual data files live in MinIO/S3. Git stores only tiny .dvc pointer files.

dvc-data bucket structure:
  nayak-mlops/
    house-prediction-model/
      v1/
        train.parquet    ← 16,512 rows, 9 columns (reference data)
        test.parquet     ← 4,128 rows
        full.parquet     ← 20,640 rows (complete dataset)

DVC pointer file stored in Git (tiny, ~100 bytes):
  data/train.parquet.dvc → {md5: "abc123...", path: "nayak-mlops/..."}

5 Whys — Why Do We Need This?

WHY 1
We need to reproduce any training run exactly, any time.
WHY 2
Because a model trained last month might behave differently today if data changed silently.
WHY 3
Because we can't commit large data files to Git — they'd bloat the repository to gigabytes.
WHY 4
Because without data versioning, we can't link a specific model to the exact data that produced it.
WHY 5
Because production audits, debugging regressions, and regulatory compliance all require proof of what data the model trained on.

⚠ Training-Serving Skew Risk
AveRooms, AveBedrms, AveOccup are derived (divided) features. If serving code computes them differently from training code, the model receives wrong inputs silently. Always use shared feature computation code.

5 Whys

WHY 1
Raw data often isn't in the format a model can use directly.
WHY 2
Models need numeric, scaled, normalized inputs — raw strings, nulls, and unscaled values degrade performance.
WHY 3
Derived features (AveRooms = TotalRooms/Households) capture richer relationships than raw totals.
WHY 4
Feature engineering is where domain knowledge enters the model — it can improve accuracy more than tuning algorithms.
WHY 5
The same features must be computed identically at training and serving time — inconsistency causes training-serving skew, the #1 silent model killer.

FastAPI Custom Predictor
main.py + logger.py — the serving code you own completely

Request Lifecycle

POST /predict {"features": [8.32, 41.0, 6.98, 1.02, 322, 2.55, 37.88, -122.23]}
  ↓ Pydantic validates: must be list of 8 floats
  ↓ arr = np.array(features).reshape(1,-1)
  ↓ prediction = model.predict(arr)[0]   ← in-RAM model, ~1ms
  ↓ BackgroundTask: log_prediction(features, prediction)
  ↓ Response: {"prediction": 4.336, "version": 1}   ← ~5ms total
     [user never waits for MinIO write]
         ↓ (after response sent)
         logger accumulates in buffer
         at 50 rows → parquet → MinIO   ← ~100ms, user unaffected

Why Background Tasks (Critical Interview Concept)

Without BackgroundTasksWith BackgroundTasks
predict [5ms] + MinIO write [100ms] = 105ms user waitpredict [5ms] = 5ms user wait. MinIO write happens after.
WHY 1
We need to log every prediction for drift detection.
WHY 2
The built-in sklearn runtime is a black box — we can't inject logging into it.
WHY 3
Evidently needs prediction logs to compare against training reference data.
WHY 4
Without prediction logs, we can't detect if the model is receiving unusual or drifted inputs.
WHY 5
Silent data drift without detection leads to business decisions made on wrong predictions — discovered only when it's too late.
# main.py pattern
@asynccontextmanager
async def lifespan(app):
    model = mlflow.sklearn.load_model("models:/best_random_forest_model/1")
    app.state.model = model
    yield
    force_flush()   # save remaining predictions on shutdown

app = FastAPI(lifespan=lifespan)

@app.post("/predict")
async def predict(req: PredictRequest, bt: BackgroundTasks):
    arr = np.array(req.features).reshape(1,-1)
    pred = app.state.model.predict(arr)[0]
    bt.add_task(log_prediction, req.features, float(pred))
    return {"prediction": pred}

Kourier — Knative Ingress Gateway
Routes external traffic to KServe pods by Host header

What Kourier Does

Kourier is a lightweight Envoy-based ingress gateway designed specifically for Knative. It receives all inbound traffic at the LoadBalancer IP and routes it to the correct Knative Revision based on the HTTP Host header.

Request flow:
  Client → DNS (Pi-hole) → 192.168.1.240 (Kourier)
  Kourier reads Host: mlflow-model-predictor.ml.ml.local
  Looks up Knative Route for this hostname
  Forwards to: mlflow-model-predictor-00001-private service
  Pod receives request on port 8080 (queue-proxy sidecar first)

Kourier vs Traefik (both running in your stack)

ComponentIPRoutes
Kourier192.168.1.240KServe/Knative services (*.ml.ml.local)
Traefik192.168.1.241Everything else (grafana, prometheus, evidently)
WHY 1
Knative needs a network layer to route traffic to the correct service revision.
WHY 2
Standard K8s ingress (Nginx/Traefik) doesn't understand Knative routing concepts like traffic splits and revisions.
WHY 3
Kourier is the lightest option — just Envoy proxy, no extra control plane. Istio would work too but adds significant overhead.
WHY 4
Canary deployments work via Kourier's traffic weight routing — 10% to revision-v2, 90% to revision-v1.
WHY 5
MetalLB gives Kourier a real LoadBalancer IP (192.168.1.240) — on-premise equivalent of cloud LoadBalancer.
DNS resolution:
  mlflow-model-predictor.ml.ml.local → 192.168.1.240 (Kourier)
  (configured in Pi-hole dnsmasq wildcard: *.ml.ml.local → 192.168.1.240)

Prediction Logger (logger.py)
Thread-safe async batch writer — captures every prediction without blocking users

Internal Flow

Request → predict() → response returned → BackgroundTask fires
                                                ↓
                                 log_prediction(features, prediction)
                                                ↓
                                 with lock:  buffer.append(row)
                                             if len >= 50: _flush()
                                                ↓ (at 50 rows)
                                 DataFrame → parquet (in-memory BytesIO)
                                 → MinIO PUT predictions/model/v1/date/batch_N.parquet

Why Threading Lock

FastAPI can handle concurrent requests with multiple threads.
Two requests arriving simultaneously can corrupt the buffer:
  Thread-1: buffer.append(row_A)  ←  race condition
  Thread-2: buffer.append(row_B)  ←  data corruption!

with threading.Lock():
  Only one thread modifies buffer at a time → safe.
WHY 1
We need a record of every prediction for drift detection and model monitoring.
WHY 2
Drift detection requires comparing production data distribution to training data distribution.
WHY 3
Without prediction logs, Evidently has no production data to compare against reference.
WHY 4
We batch 50 rows instead of writing per-request to reduce MinIO PUT operations by 50x and avoid adding latency.
WHY 5
Prediction logs are the audit trail — in regulated industries, every prediction must be provable. Parquet in S3 is queryable forever.
BATCH_SIZE = 50
FEATURES = ["MedInc","HouseAge","AveRooms","AveBedrms",
            "Population","AveOccup","Latitude","Longitude"]
buffer = []
lock = threading.Lock()

def log_prediction(features, prediction):
    row = dict(zip(FEATURES, features))
    row.update({"prediction": prediction,
                "timestamp":  datetime.utcnow().isoformat(),
                "model_version": "1"})
    with lock:
        buffer.append(row)
        if len(buffer) >= BATCH_SIZE:
            _flush()

def _flush():
    if not buffer: return
    df  = pd.DataFrame(buffer)
    buf = io.BytesIO()
    df.to_parquet(buf, index=False)
    buf.seek(0)
    key = f"predictions/mlflow-model/v1/{date.today()}/batch_{counter}.parquet"
    s3.put_object(Bucket="predictions", Key=key, Body=buf.getvalue())
    buffer.clear()

The Problem It Solves

Prometheus works by PULLING metrics from running services every 30s.
But Evidently CronJob:
  → runs for 30 seconds
  → calculates drift
  → exits (pod is gone)
  
Prometheus can't scrape a pod that no longer exists.

  CronJob pushes metrics to Pushgateway BEFORE exiting.

                          Prometheus PULLS every 30s

Architecture · ServingServing Flow — Single Request Lifecycle

What happens from client curl to prediction response in ~5ms.

SERVING FLOW — ONE REQUEST

Client:
  curl -X POST http://mlflow-model-predictor.ml.ml.local/predict
       -d '{"features": [8.3252, 41.0, 6.98, 1.02, 322.0, 2.55, 37.88, -122.23]}'

Step 1: DNS Resolution
  mlflow-model-predictor.ml.ml.local
  → Pi-hole dnsmasq wildcard *.ml.ml.local
  → resolves to 192.168.1.240 (Kourier LoadBalancer IP)

Step 2: Kourier Routing
  Request arrives at Kourier port 80
  Reads Host header: mlflow-model-predictor.ml.ml.local
  Looks up Knative Route for this hostname
  Forwards to: mlflow-model-predictor-00001-private:80

Step 3: Queue-Proxy Sidecar
  First container in pod (injected by Knative)
  Handles: autoscaling signals, concurrency limits, metrics
  Forwards to main predictor container at port 8080

Step 4: FastAPI Predictor
  Receives POST /predict
  Pydantic validates: features must be list of 8 numbers
  arr = np.array(features).reshape(1, -1)
  prediction = model.predict(arr)[0]    ← in-RAM RandomForest, ~1ms
  Registers BackgroundTask: log_prediction(features, pred)
  Returns: {"prediction": 4.336, "model": "...", "version": 1}

Step 5: Response travels back
  FastAPI → Queue-Proxy → Kourier → DNS → Client
  Total time: ~5–15ms

Step 6: Background (AFTER response sent, user doesn't wait)
  logger.log_prediction(features=[8.3252,...], prediction=4.336)
  Appends row to in-memory buffer (thread-safe)
  If buffer ≥ 50: convert to parquet → write to MinIO (100ms)
  User never sees this delay.

Metrics · RegressionRegression Metrics — Full Reference

Your model predicts a continuous number (house price). Click any card for formula, when to use, when NOT to use, what to do when it degrades.

📏
RMSE
REGRESSION
Root Mean Square Error
√( mean( (actual − predicted)² ) )
UnitSame as target. If predicting $100k units, RMSE=0.52 → avg error ≈$52k
Your modelRMSE ≈ 0.52 (target range 0.5–5.0, so ~25% relative error)
SensitivitySquaring amplifies large errors — one very wrong prediction hurts RMSE a lot
Use whenLarge errors are especially costly (financial, medical, safety)
Don't useWhen outliers dominate and you don't want them to drive the metric
If it rises1) Check data drift 2) Check recent deployment 3) Check feature distributions 4) Consider retraining
vs MAERMSE ≫ MAE means a few large errors. RMSE ≈ MAE means errors are uniformly distributed.
📐
MAE
REGRESSION
Mean Absolute Error
mean( |actual − predicted| )
UnitSame as target — directly interpretable by non-technical stakeholders
Your modelMAE ≈ 0.37 → median absolute error $37k
vs RMSEMAE treats all errors equally. RMSE punishes large errors more heavily.
Use whenAll errors matter equally, no special penalty for large errors
Don't useWhen large errors have disproportionate business impact
GradientNot differentiable at zero — use MSE as training loss, MAE for evaluation
📊
R² Score
REGRESSION
Coefficient of Determination
1 − (SS_res / SS_tot)
Scale−∞ to 1.0. Negative means worse than predicting the mean.
R²=1.0Perfect — model explains 100% of variance
R²=0.81Your model: explains 81% of house price variance
R²=0.0Model is as good as always predicting the mean — useless

Metrics · ClassificationClassification Metrics — Precision, Recall, F1

For fraud, churn, spam, disease — anything predicting categories. Everything derives from the Confusion Matrix.

Confusion Matrix — The Foundation

                  Predicted YES      Predicted NO
Actual YES    |  True Positive (TP) | False Negative (FN)  |  ← Missed
Actual NO     | False Positive (FP) |  True Negative (TN)  |  ← False alarm

Cost analysis:
  FP = false alarm  → use Precision to minimize FPs
  FN = missed real  → use Recall to minimize FNs
  
Examples:
  Spam: FP=important email blocked(costly) → maximize Precision
  Fraud: FN=fraud missed(costly)           → maximize Recall
  Disease: FN=cancer missed(deadly)        → maximize Recall
🎯
Precision
CLASSIFICATION
"When I say YES, am I right?"
TP / (TP + FP)
P=1.0Every YES prediction was correct — zero false alarms
P=0.5Half your YES predictions were wrong — 50% false alarm rate
Use whenFalse positives are costly: spam (blocked legit email), recommendation (showing irrelevant items)
Trade-offHigher precision → lower recall. You miss more real positives to avoid false alarms.
If it dropsModel flagging too many false positives → users annoyed, costs rising, trust eroding
🔍
Recall
CLASSIFICATION
"Of all actual YES, how many caught?"
TP / (TP + FN)
Also calledSensitivity, True Positive Rate, Hit Rate
R=1.0Caught every single positive — zero misses
R=0.5Missed half the actual positives
Use whenFalse negatives are costly: fraud (missed = financial loss), cancer (missed = death)
Trade-offHigher recall → lower precision. More false alarms to catch more real positives.
If it dropsReal positives getting through undetected → fraud escaping, patients not caught
⚖️
F1 Score
CLASSIFICATION
Harmonic mean of Precision & Recall
2 × (P × R) / (P + R)
Why harmonicArithmetic mean rewards imbalance (P=1.0,R=0.0 → mean=0.5, looks ok). Harmonic punishes it: F1=0.0
F1=1.0Perfect precision AND perfect recall
F1=0.0Either precision or recall is zero — useless model
Use whenBoth FP and FN have significant cost — churn, content moderation
ImbalancedUse F1 instead of accuracy — accuracy is misleading when one class dominates
F-betaF2 weights recall 2x. F0.5 weights precision 2x. Use when one matters more than other.
✅
Accuracy
CLASSIFICATION
% of all predictions correct
(TP + TN) / (TP + TN + FP + FN)
Use whenBalanced classes AND all errors cost the same
The trap99% legit transactions, 1% fraud → 99% accuracy by saying everything is legit. Useless model.
RuleIf class imbalance > 10:1, never use accuracy. Use F1, Recall, or AUC-ROC.

AUC-ROC

Drift · All Types6 Drift Types — Click to Expand

Drift = when the statistical world your model learned from diverges from the world it's predicting in. 6 distinct types, each with different causes, detection, and fix.

TYPE 1
Data Drift
Covariate ShiftP(X) changes
Input feature distributions change. The relationship P(Y|X) stays the same.
What It Is

Production inputs have different statistical properties from training inputs. The model's learned relationships may still be correct but it's being applied to data it wasn't optimized for.

Real Example
Training (2020): avg MedInc = 3.8 (working-class neighborhoods)
Production (2026): avg MedInc = 7.2 (app now popular in wealthy areas)

Input distribution shifted. Model may underestimate prices
for wealthy neighborhoods it rarely saw in training.
Detection
  • PSI per feature — PSI > 0.2 = significant drift
  • KS test — p-value < 0.05 = distributions differ
  • Evidently DataDriftPreset does both automatically
What To Do
  1. Confirm drift is real (not a data pipeline bug)

WHY 1
Production predictions are getting less accurate over time.
WHY 2

WHY 3
The world changes — user demographics shift, new markets open, economic conditions evolve.

TYPE 2
Concept Drift
P(Y|X) changesHardest to detect
The relationship between features and outcome changes. Same inputs → different correct outputs.
What It Is

The most dangerous drift. Feature distributions can look perfectly normal while model performance silently degrades because the world's rules have changed.

#1 Silent KillerNot Technically "Drift"

Production input: AveOccup = 0.89

  • Evidently compares serving features vs training features
  • Integration test: known input → assert serving features == training features

  • Feature Store (Feast) — one code path computes features for both training and serving
  • Shared library — feature computation in a pip package imported by both train.py and predictor
  • Automated tests — CI/CD test that verifies serving features match training features for known inputs

Drift · StatisticsPSI & KS Test — Deep Dive

PSI — Population Stability Index

Formula: PSI = Σ (Actual% − Expected%) × ln(Actual% / Expected%)

Step by step:
  1. Bin the feature into 10 equal-width buckets
  2. Expected%: % of TRAINING data in each bucket
  3. Actual%:   % of PRODUCTION data in each bucket
  4. Per bucket: (Actual% − Expected%) × ln(Actual%/Expected%)
  5. Sum all buckets = PSI

Interpretation:
  PSI < 0.10   → STABLE. No action needed.
  PSI 0.1–0.2  → SLIGHT change. Monitor.
  PSI > 0.20   → SIGNIFICANT drift. Investigate.
  PSI > 0.25   → MAJOR drift. Model likely impacted.

Industry standard: Finance / credit scoring use PSI routinely.
Regulators require PSI reports quarterly. PSI > 0.25 triggers mandatory model review.

KS Test — Kolmogorov-Smirnov

What it measures: Maximum vertical distance between two CDFs

How it works:
  1. Sort training and production feature values separately
  2. Compute CDF (Cumulative Distribution Function) for each
  3. Find maximum vertical distance between the two CDFs → D statistic
  4. Calculate p-value from D and sample sizes

Interpretation:
  D = 0    → identical distributions
  D = 1    → completely different distributions
  p < 0.05 → distributions are significantly different (drift detected)
  p > 0.05 → no statistically significant difference

Advantage: Non-parametric — makes no assumptions about distribution shape.
Works on small samples. Shows WHERE maximum difference occurs.

Components · Deep DiveKnative — Serverless Platform

What Knative Adds To K8s

Plain K8s: define Deployment + Service + HPA manually
Knative:   define "Service", it manages everything:
  → Knative Service → Configuration → Revision (immutable snapshot)
                   → Route (traffic rules)

Each code change = new Revision. Traffic can split across Revisions:
  Revision 3: 90% of traffic (stable)
  Revision 4: 10% of traffic (canary)
  
KServe uses Knative Revisions as the canary mechanism.

Scale to Zero Flow

minReplicas=0 (default):
  No traffic for 30s → pods terminated → cost savings

New request arrives:
  1. Kourier receives request
  2. Knative activator buffers it (client waits)
  3. Controller starts new pod (~30-60s cold start for ML models)
  4. Pod loads model from MLflow/MinIO
  5. Activator forwards buffered request

For production with SLA (use minReplicas=1):
  → Always one warm pod, no cold start, predictable latency
  → Pay for always-on pod but guarantee response time

Components · Deep DivePrometheus Pushgateway

Why It Exists

Prometheus model: PULL (scrapes running services every 30s)

Problem with CronJobs:
  - Job runs for 20 seconds
  - Job exits (pod gone)
  - Prometheus scrapes 30s later — nothing to scrape!

Pushgateway solution:
  - CronJob pushes metrics to Pushgateway before exiting
  - Pushgateway keeps last pushed value (persistent)
  - Prometheus scrapes Pushgateway every 30s normally

Metrics Lifecycle

T=0:00  Evidently CronJob starts
T=0:25  drift_check.py pushes metrics to Pushgateway
          ml_drift_detected=1, ml_drift_share=0.5
T=0:26  CronJob pod exits
T=0:30  Prometheus scrapes Pushgateway → stores metrics
T=1:00  Grafana queries Prometheus → shows drift alert
T=1:01  Next CronJob starts...

Metrics persist in Pushgateway until overwritten by next job.
Even if next job fails, last known values remain visible.

TerminologyComplete MLOps Dictionary

Every term you'll hear in interviews, documentation, and production discussions.

Artifact StoreINFRA
Storage backend for MLflow artifacts — model files, plots, data. Separate from metadata (PostgreSQL). In your stack: MinIO mlflow-artifacts bucket.
mlflow.log_model() uploads model.pkl to s3://mlflow-artifacts/1/run_id/artifacts/model/
Background TaskOPS
FastAPI mechanism to run code after response is sent. The user gets their prediction before MinIO write completes. Enables async logging without latency impact.
background_tasks.add_task(log_prediction, features, pred) — executes after {"prediction": 4.336} is returned.
Canary DeploymentOPS
Pattern where new model version gets small % of traffic first. Validate before full rollout. KServe: canaryTrafficPercent: 10. Rollback: set to 0.
New model v2 gets 10% traffic for 30 minutes. Error rate stays low. Promote to 100%.
Concept DriftML
P(Y|X) changes — the relationship between features and outcome shifts. Most dangerous drift: features look normal but model performance degrades. Requires ground truth to detect.
Pre-COVID: high income + city → high price. Post-COVID: high income + city → lower price (remote work shifted demand). Same features, wrong predictions.
Confusion MatrixML
2×2 table: TP (correct YES), TN (correct NO), FP (false alarm), FN (miss). Foundation for precision, recall, F1. Row = actual, column = predicted.
Cancer screening: FN (missed cancer) costs more than FP (unnecessary biopsy) → optimize for Recall.
Covariate ShiftML
Synonym for Data Drift. P(X) changes — input feature distributions shift while the relationship P(Y|X) stays the same. Detectable with PSI/KS tests.
Training on 1990s houses, serving requests for new 2024 construction — HouseAge distribution shifts completely.
CronJob (Kubernetes)INFRA
K8s resource that creates Jobs on a cron schedule. Pod runs, completes task, exits. Used for Evidently drift detection (hourly), lifecycle cleanup, batch scoring.
schedule: "0 * * * *" creates a new pod every hour, runs drift_check.py, then pod is deleted automatically.
Data DriftML
P(X) changes — input feature distributions in production differ from training distributions. P(Y|X) unchanged. Most common drift type. Detected by PSI and KS tests.
MedInc training avg=3.8, production avg=7.2. PSI=0.45 (major drift). Evidently flags as drifted.
DVC (Data Version Control)DATA
Git for data. Stores large files (parquet, CSV) in S3/MinIO, keeps tiny pointer files (.dvc) in Git. Links code version to exact data version for reproducibility.
dvc add train.parquet creates train.parquet.dvc (100 bytes in Git). dvc push uploads 150MB parquet to MinIO.
EvidentlyOPS
Open source ML observability library + UI server. Compares reference data (training) against current data (production). Runs PSI/KS tests, generates HTML reports and Prometheus metrics.
Evidently CronJob reads train.parquet + today's prediction parquets, runs DataDriftPreset, pushes ml_drift_detected=1 to Pushgateway.
F1 ScoreML
Harmonic mean of Precision and Recall. Punishes imbalance between the two. Preferred over accuracy for imbalanced datasets. Range 0.0 to 1.0.
P=0.9, R=0.7 → F1 = 2×(0.9×0.7)/(0.9+0.7) = 0.788. Better than using either alone.
Feature DriftML
Drift in specific individual features while others remain stable. Evidently reports PSI per column, enabling pinpoint identification of which features changed and why.
Population PSI=0.45 (major), all others stable → likely data pipeline bug computing Population, not real-world change.
Feature StoreDATA
Centralized system for computing, storing, and serving features. Eliminates training-serving skew by using same computation code for both. Has offline store (batch training) and online store (low-latency serving).
Feast: AveOccup computed once, stored in both offline (training) and online (Redis for serving) stores. No skew possible.
GrafanaINFRA
Visualization and alerting layer on top of Prometheus. Queries PromQL, renders dashboards, sends alerts via Slack/email/PagerDuty. Does not store data itself.
Grafana drift dashboard: ml_drift_detected stat panel (Red/Green), ml_drift_share gauge, request rate time series — all from Prometheus queries.
Ground Truth DelayML
Time between making a prediction and knowing the correct answer. Makes real-time performance monitoring impossible without proxy metrics or drift detection.
House price model predicts $450k today. Actual sale happens 45 days later. Can't calculate RMSE until then. Solution: drift monitoring (immediate) + delayed evaluation pipeline.
InferenceServiceINFRA
KServe Custom Resource Definition. Declares a model serving deployment. Creates pods, services, Knative routes, HPA automatically from a single YAML spec.
kubectl apply -f inference-service.yaml → creates predictor pod, Knative Service, Kourier Route all automatically.
IRSA (IAM Roles for Service Accounts)INFRA
AWS mechanism linking K8s ServiceAccounts to IAM roles. Pods assume roles via token projection — no access keys anywhere. Credentials rotate automatically.
predictor-sa → IAM role AllowPredictionsWrite. Pod writes to S3 without any credentials in YAML. Zero secrets to rotate or leak.
KnativeINFRA
Kubernetes serverless framework. Each deployment creates immutable Revisions. Traffic splits across Revisions for canary. Scale-to-zero when idle.
KServe uses Knative Revisions as the canary mechanism. canaryTrafficPercent: 10 routes 10% to Revision-v2, 90% to Revision-v1.
KourierINFRA
Lightweight Envoy-based ingress for Knative. Routes traffic to correct Knative Service by Host header. Lighter alternative to Istio. MetalLB gives it a real IP.
Kourier at 192.168.1.240. Routes mlflow-model-predictor.ml.ml.local to KServe predictor pod via Knative Route lookup.
KS Test (Kolmogorov-Smirnov)ML
Non-parametric test measuring max difference between two CDFs. p < 0.05 means distributions are significantly different. Default test for numerical features in Evidently.
Training MedInc vs production MedInc CDFs. D=0.45, p=0.001 → drift detected (p < 0.05).
KServeINFRA
K8s-native ML model serving. Knative-based autoscaling, traffic splitting for canary, multi-framework support. Standard for ML serving on Kubernetes.
InferenceService YAML → KServe creates FastAPI predictor pod with autoscaling, health checks, Kourier routing, and /metrics endpoint.
Label DriftML
P(Y) changes — target variable distribution shifts. For classification: class frequencies change. Model calibration breaks when base rate changes significantly.
Training: 1% fraud. Production: 8% fraud (attack wave). Model underestimates fraud. Fix: adjust threshold or retrain.
Lifecycle Policy (S3/MinIO)DATA
Rules that automatically delete/transition objects after specified time. Essential for cost control — predictions bucket grows unbounded without it.
mc ilm add local/predictions --expiry-days 90 → all prediction parquets deleted automatically after 90 days.
MAE (Mean Absolute Error)ML
Average of absolute differences between predictions and actuals. Same unit as target. Treats all errors equally. More interpretable than RMSE for business stakeholders.
Your model MAE ≈ 0.37 → median absolute error $37,000. Compared to mean house price ~$207k → ~18% typical error.
MinIOINFRA
Open-source S3-compatible object storage. Same API as AWS S3. Switch to cloud by changing endpoint_url only — zero code changes. Your on-premise data lake.
boto3.client("s3", endpoint_url="http://192.168.1.15:30900") — identical to AWS S3 except endpoint URL.
MLflowOPS
Open source ML lifecycle platform. 4 components: Tracking (log experiments), Artifact Store (store files), Model Registry (version management), Serving (you replaced with KServe).
mlflow.log_metric("r2", 0.81) → PostgreSQL. mlflow.log_model() → MinIO pkl. mlflow.register_model() → Registry v1.
Model RegistryOPS
Centralized catalog of model versions with lifecycle stages: None → Staging → Production → Archived. Source of truth for what's deployed. KServe loads models by registry reference.
models:/best_random_forest_model/1 — KServe loads this reference → MLflow resolves → MinIO pkl download.
Network Policy (K8s)INFRA
K8s resource controlling pod-to-pod traffic. Default deny-all, explicitly allow needed connections. Limits blast radius if a pod is compromised.
Predictor pod policy: allow egress to MinIO:9000 and MLflow:5000 only. All other outbound blocked.
Object StorageINFRA
Flat key-value storage for large unstructured data (bucket/key → binary). No filesystem hierarchy. HTTP API. Petabyte-scale. 11 nines durability on AWS S3.
s3://mlflow-artifacts/1/run_id/artifacts/model/model.pkl — looks like a path, but "1/run_id/artifacts/model/model.pkl" is the key in a flat namespace.
PSI (Population Stability Index)ML
Measures distribution shift per feature. Bins data, compares % per bucket. PSI < 0.1 stable, 0.1-0.2 slight, >0.2 significant, >0.25 major. Finance industry standard.
MedInc: 30% of training in [0-4] bucket vs 10% in production → PSI = 0.35 (significant drift).
Prediction DriftML
Model output distribution changes over time. Can occur without feature drift. Often caused by serving code bugs. Monitor prediction mean and std in Prometheus/Grafana.
All predictions = 4.336 (std ≈ 0). Your test sent identical requests 55 times. In production indicates serving bug.
PrometheusINFRA
Pull-based time series metrics database. Scrapes /metrics endpoints every 30s. Stores with labels. PromQL for queries. K8s monitoring standard. Foundation for Grafana dashboards.
Every 30s: Prometheus → GET kserve-pod:8080/metrics → stores http_requests_total, latency_seconds. Grafana queries via PromQL.
Prometheus PushgatewayINFRA
Intermediary for short-lived jobs (CronJobs) to push metrics to Prometheus. Jobs push before exiting. Prometheus scrapes Pushgateway normally. Persists last-pushed value.
Evidently CronJob runs 20s, pushes ml_drift_detected=1 to Pushgateway, exits. Prometheus scrapes Pushgateway at :30s and :60s. Grafana shows alert.
R² (R-Squared)ML
Proportion of target variance explained by the model. Range −∞ to 1.0. R²=0.81 means model explains 81% of variance. R²=0 = predicting the mean. R²<0 = worse than mean.
Your model R²=0.81. Baseline (always predict mean) R²=0. Your model is 81% better than the naive baseline.
RawDeployment (KServe)INFRA
KServe mode using standard K8s Deployment instead of Knative. Works with local/private registry images. No scale-to-zero, no built-in canary. Good for development phase.
annotation: serving.kserve.io/deploymentMode: RawDeployment — used when Knative can't resolve local containerd image.
Recall (Sensitivity)ML
Of all actual positives, how many did the model catch? TP/(TP+FN). Maximize when false negatives are costly (fraud, disease, safety).
Cancer screening Recall=0.99 → catches 99% of actual cancer cases. Missing 1% (FN) could be fatal.
RMSE (Root Mean Square Error)ML
√(mean squared errors). Same unit as target. Penalizes large errors heavily (squaring). Your model: RMSE ≈ 0.52 (target in $100k units = ~$52k average error).
RMSE 0.52 vs MAE 0.37 → RMSE > MAE suggests some predictions have large errors. Most are close, a few are very wrong.
ServiceMonitorINFRA
Prometheus Operator CRD telling Prometheus which K8s services to scrape. Declarative — add a YAML file, Prometheus auto-discovers. No prometheus.yaml editing needed.
ServiceMonitor selector matchLabels: app=mlflow-model → Prometheus scrapes all matching pods every 30s automatically.
Shadow ModeOPS
New model runs in parallel but its output is NOT served to users. Log both models' outputs. Validate new model behavior before canary.
Old model → user response. New model → log only. After 1 week: compare logs to validate new model before exposing to users.
Training-Serving SkewML
#1 silent model killer. Feature computation differs between training and serving code. Same raw input → different feature values → model sees wrong data. No errors, just wrong predictions.
AveOccup = Pop/Households (training) vs Pop/Bedrooms (serving). Both "reasonable" but model gets wrong value. Silent degradation for months.

Open source vulnerability scanner for container images and dependencies. Used in CI/CD to block deployment of images with known CVEs. aquasecurity/trivy-action GitHub Action.
